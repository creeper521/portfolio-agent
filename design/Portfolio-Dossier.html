N° 01 · 工程交付

SQL 审计 / 排查工具

面向远程 SQL 日志的查询、路由、归档与导出工具。支持多目标多来源配置路由、异步任务、时间倒序展示、Excel 导出、结果归档复用与软删除，以及 WebSocket 实时推送与轮询兜底。

完成状态◑实现 · 已测试
个人贡献独立完成 · INDEPENDENT
时间2026.05.29 — 06.05
证据7 项 · 经脱敏
技术栈
Spring BootWebSocket异步任务 Excel 导出归档复用轮询兜底
背景Background

线上问题排查依赖远程 SQL 日志，但日志查询散落在不同环境与目标库中，排查一次问题需要在多个入口反复手动查询、复制、比对，效率低且容易遗漏命中记录。

问题Problem
  • 查询入口分散，多环境、多目标缺乏统一路由；
  • 同步查询阻塞，单次大日志查询拖慢整个排查；
  • 历史查询结果无法复用，重复劳动；
  • 命中 SQL 难以完整保留与归档导出。
个人职责Role

独立完成该工具的需求分析、方案设计、实现、自测与使用文档。属个人独立交付的工程产出（INDEPENDENT）。

方案Approach

采用异步任务承接远程 SQL 日志查询，避免长耗时查询阻塞主流程；多环境、多目标拆为三类配置路由，按来源与目标分发；查询结果归档复用并支持软删除；命中 SQL 以时间倒序完整展示，支持 Excel 导出。

实时性方面，以 WebSocket 推送任务进度，并以轮询作为兜底，确保连接异常时仍能拿到结果。

客户端请求 → 异步任务调度 → 多目标路由 → 远程 SQL 日志查询 → 归档 → WebSocket / 轮询回传 → 时间倒序展示 / Excel 导出
难点Challenges
  • 异步与实时性的权衡：既要查询不阻塞，又要让用户感知进度，因此引入 WebSocket + 轮询双通道；
  • 多目标路由的可维护性：环境、来源、目标组合多，需拆成可配置的三类路由避免硬编码；
  • 结果归档与复用：相同查询不应重复执行，需设计归档键与软删除策略。
验证Verification

完成连续上传不同语言的核心场景自测，并产出使用说明与验收文档。验证证据等级为 VERIFIED（有自测与验收记录）。

说明：本工具为内部排查工具，"实现并测试"指完成实现与自测；是否纳入正式上线流程以团队最终确认为准，此处不表述为"已正式上线交付"。

结果Outcome

工具整合了分散的远程 SQL 日志查询入口，支持异步、路由、归档、导出与实时回传，显著降低重复排查成本。产出包含完整的使用说明、验收文档、打包与备份记录。

未完成项Limitations
  • 正式上线 / 纳入团队工具链的流程尚未最终确认；
  • 部分边界场景的测试覆盖可进一步加强。

状态判定依据：有实现记录与自测/验收证据，但未确认正式提交、验收或上线，故标记为「实现 · 已测试」（IMPLEMENTED_TESTED），不得表述为「已正式交付」。

反思Reflection

异步 + 兜底的双通道设计让我更理解"可用性不能完全依赖单一通道"的工程直觉；多目标路由的可维护性也让我体会到配置驱动相对于硬编码的长期价值。
