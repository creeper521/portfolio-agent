```text

项目文档目录：D:\code\agent\docs
实习知识库目录：C:\Users\WIN10\Documents\杂项\实习学习-Obsidian
```

知识库目录是私有端的原始数据源，包含全部日报、专题文档和附件。该绝对路径只用于当前开发环境说明，后续代码不得硬编码，应通过环境变量或私有配置注入。

## 3. 项目背景

项目发起人在 Java 游戏服务端实习期间，持续使用 Markdown 记录每日任务、线上问题排查、项目源码学习、内部工具开发、AI 工程实验和容器化学习。

当前原始知识库位于本地 Obsidian 目录，主要时间范围为 2026 年 4 月至 7 月。截至 2026-07-14，知识资产大致包括：

- 105 篇 Markdown；
- 73 篇日常任务与产出记录；
- 32 篇专题、设计、攻略或学习文档；
- 约 697 KB 文本；
- 约 350 个迁移附件；
- 355 处图片引用；
- 大量真实的需求分析、调试过程、SQL、架构图、测试记录和反思。

统计口径：原语雀迁移报告记录 104 篇正文；本地另有 1 篇索引，共 105 篇 Markdown。日报和专题数量、文本大小与图片引用由 2026-07-14 对知识库目录的本地文件扫描得到。后续同步流程应生成版本化统计结果，避免人工数字与知识库增量发生漂移。

知识库已能支持人工回顾，但还不适合直接公开，也不适合仅依赖普通语义检索进行准确问答。

## 5. 当前问题

### 5.1 零散日报不等于可复用的项目知识

同一个项目可能跨多天迭代。例如 SQL 审计/排查工具横跨 5 月 29 日至 6 月 5 日，信息分布在需求、日报、使用攻略、验收文档和截图中。只检索单篇笔记无法得到完整项目故事。

### 5.2 普通 RAG 容易把计划误判为成果

原始笔记中同时存在：

- “接下来准备实现”；
- “待自测”；
- “实现完成”；
- “自测没问题但暂不提交”；
- “代码提交并打包”；
- “仅学习、跟进或评估”。

如果仅按语义相似度召回片段，模型可能把计划、原型、学习和正式交付混为一谈。

### 5.3 原始知识库不能直接公开

知识库包含或可能包含：

- 内部服务器信息；
- 数据库表和 SQL；
- 账号、密码、API Key、Token；
- 玩家或测试账号标识；
- 内部链接、目录和环境名称；
- 未公开的业务流程；
- 截图中的敏感信息。

公开服务不能直接连接原始 Obsidian 目录，也不能只依赖系统提示词保护隐私。

### 5.4 纯聊天界面不利于建立信任

面试官和导师需要快速浏览代表性项目、完成状态、时间线和证据。只有聊天框会导致：

- 访问者不知道该问什么；
- 重要项目不容易被发现；
- 回答缺少可验证依据；
- 产品看起来更像普通 RAG Demo，而不是作品集。

### 5.5 私有检索与公开展示的目标不同

公开端追求清晰、可信、脱敏和叙事质量；私有端追求完整、可追溯、可检索和持续更新。两者不能共用同一个无边界索引。

## 6. 产品愿景

长期目标是建设一套共享领域模型、但数据与部署严格隔离的双端系统。V1 不同时开发两个完整应用：先完成公开 Portfolio Agent；私有查询、归纳、脱敏、面试复盘和快照准备由 Codex、Skill、确定性脚本与人工审核承担，验证价值后再决定是否产品化为 Internship Copilot。

### 6.1 公开 Portfolio Agent

面向技术面试官、实习导师、HR 和普通访客，以交互式作品集形式回答：

- 实习期间做过什么；
- 核心工程产出；
- 个人职责和技术难点；
- 方案演进与取舍；
- 验证结果；
- 能力成长；
- 未完成项和反思。

公开 Agent 只能读取人工审核并发布的脱敏内容。

### 6.2 私有 Internship Copilot

面向本人，读取完整本地知识库，用于：

- 查询全部日报和专题；
- 原文追溯；
- 项目聚合；
- 查找相似排障案例；
- 生成日报、周报、月报、述职和面试材料；
- 检查知识库质量和敏感信息；
- 生成待审核的公开内容候选。

V1 不建设独立的私有 Web、CLI Agent、Spring Boot 应用或私有数据库。上述能力通过 Codex、Skill、`rg`/文件工具、确定性脚本、JSON Schema 和人工审核流程提供；Codex 可以读取完整知识库，但只能只读处理，不自动修改原始笔记。正式私有 Copilot 延后到 V2。

## 7. 核心产品原则

### 7.1 事实优先于语言表现

模型可以调整表达方式，但不能决定事实是否存在。项目、结论、完成状态和证据应由结构化事实库控制。

### 7.2 结构化事实为主，RAG 为辅

结构化层负责回答“做过什么、是否完成、有什么证据”。V1 公开端使用 FAQ、别名和轻量内存检索补充细节；全文或向量检索只作为未来私有端的候选能力，并且必须由评测证明有必要。

### 7.3 不把计划当成果

所有成果都必须标记状态，例如：

- `COMPLETED`：完成并有充分证据；
- `IMPLEMENTED_TESTED`：实现与自测完成；
- `PROTOTYPE`：原型；
- `DESIGNED`：设计完成；
- `LEARNING`：学习或研究；
- `PLANNED`：计划；
- `UNKNOWN`：资料不足。

### 7.4 隐私在数据和工具层强制执行

公开应用不具备读取原始文件、私有索引和私有数据库的能力。提示词不是安全边界。

### 7.5 公开内容必须通过保密义务检查

删除凭据和内部地址不等于允许公开。项目名、业务流程、内部工具、表结构、部署方式和截图还需要经过雇主保密义务审查。未经确认的内容应使用通用化名称，或标记为 `REQUIRES_CONFIRMATION` / `BLOCKED`，不得进入公开快照。

### 7.6 公开内容必须经过人工审核

私有知识只能通过“提取候选 → 脱敏 → 人工审核 → 发布快照”进入公开端。Agent 不应自行决定公开内部内容。

### 7.7 证据可追溯

重要结论应关联日期、项目、证据类型和脱敏摘要。私有端还应能定位原始文件与片段。

### 7.8 每端单 Agent 优先

公开端采用单 Agent 与多个确定性工具；未来私有端产品化后也先采用单 Agent。这里的“双端”不等于多 Agent 协作，不以多 Agent、知识图谱或复杂编排增加不必要成本。

### 7.9 生成模型不可用时仍可回答

公开作品集不能把可用性完全交给生成模型。DeepSeek 超时、限流、服务异常、Tool Calling 失败或流式中断时，系统应使用 FAQ、结构化事实和确定性模板生成完整兜底回答。兜底模式仍必须遵守状态、证据、受众和保密约束。

## 8. 已确认的方向

截至 v0.3，已经确认：

- 长期目标同时包含公开作品集 Agent 和私有知识能力，但 V1 只产品化公开端；
- 公开端主要面向技术面试官和实习导师；
- 私有处理由 Codex 承担，可查询全部日报、定位原文、归纳候选事实并辅助面试复盘；
- 长期技术方向倾向 Vue 3、Spring Boot、Spring AI；PostgreSQL 仅作为 V2 私有端候选，V1 公开端不使用数据库；
- V1 只建设公开 Vue + Spring Boot 应用，私有端由 Codex + Skill/脚本 + 人工审核替代；
- V1 公开数据采用人工审核、版本化的 JSON 快照，并在应用启动时加载到内存；
- 公开问答模型使用用户提供的 DeepSeek V4 Pro API，且只能看到公开快照；
- DeepSeek 不可用时，由 FAQ + 结构化查询 + 确定性模板提供稳定兜底回答；
- 公开与私有数据、索引、工具和凭据严格分离；
- 公开端不直接访问 Obsidian；
- 私有内容通过人工审核单向发布；
- 第一版 Obsidian 只读；
- 公开网站使用真实姓名和第一人称表达，并说明内容由 AI 根据审核资料生成；
- 公司是否实名取决于保密要求，业务、项目和内部系统默认使用公开化名；
- 第一批四个核心主题为 SQL 审计/排查工具、运营后台多语言图片修复、内部测试工具角色信息清理、运营后台操作日志原型；
- 公开访问不设置访问码，使用限流和每日额度；
- 展示求职邮箱、GitHub 和简历入口；
- 代码与最终公开快照可以公开，知识库、候选稿和扫描报告不得进入公开仓库；
- Vue 构建产物由 Spring Boot 托管，以单 Docker 镜像部署到自有服务器，并由 Nginx、域名和 HTTPS 对外提供；
- 首个可访问版本按 4 至 6 周规划；
- 截图不阻塞 V1，仅在确有展示价值且通过 OCR、脱敏与保密检查时使用；
- 不建设模拟面试功能，面试复盘由私有 Codex 工作流完成；
- 第一版不做多 Agent、语音数字人、Neo4j 和 Kubernetes；
- 事实状态和证据是核心能力，不是附加功能。

## 9. 当前假设与未决问题

以下执行变量仍需在实现或发布时填写，不再改变产品方向：

1. DeepSeek V4 Pro 的实际 `baseUrl`、`modelName`、限额和 Tool Calling/SSE 兼容性；
2. 自有服务器的操作系统、资源、现有 Nginx/Docker 环境和发布权限；
3. 最终域名、求职邮箱、GitHub 和简历 URL；
4. 第一批截图是否存在不可替代的展示价值，以及逐张保密审查结果；
5. 真实公司名是否允许出现，哪些具体细节需要导师或雇主确认；
6. 公网每日模型预算和基于真实流量调整后的限流参数；
7. V2 私有 Copilot 的启动时间，继续以本地工作流频率和重复成本为依据。

## 10. 成功标准

项目成功不以“能聊天”为标准，而以以下结果衡量：

- 面试官能快速理解核心产出；
- 项目完成状态准确；
- 回答可追溯到证据；
- 技术追问能够连续深入；
- 公开端不会泄漏私有信息；
- DeepSeek 不可用时，核心问题仍能由确定性兜底引擎完整回答；
- V1 Codex 私有工作流能够完整查询日报、定位原文并生成待审核快照；
- 新增日报可以通过本地审核流程进入公开事实快照；
- 回答能够用于真实的面试、述职和复盘场景。

## 11. 给评审 AI 的重点问题

请重点评审：

1. “V1 公开应用 + 私有工具链、V2 私有应用”的分期是否合理；
2. 公开与私有数据隔离是否充分；
3. “结构化事实 + FAQ + 轻量检索”是否足以支撑 V1；
4. 哪些需求属于过度设计；
5. 哪些关键用户旅程尚未覆盖；
6. 项目状态和证据模型是否足以防止夸大；
7. 如何以更低成本完成 V1；
8. 哪些安全风险仍被低估。
