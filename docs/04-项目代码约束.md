# 实习作品集 Agent：项目代码约束

> 适用项目：`D:\code\agent`  
> 生效日期：2026-07-14  
> 适用范围：生产代码、测试代码、构建脚本和后续功能开发

## 1. 目的

本规范用于保证项目在继续扩展作品集、问答引擎和后续 DeepSeek 能力时，仍保持清晰的业务边界、显式的数据结构、稳定的错误契约和可验证的安全边界。

本规范是代码审查和发布验证的依据。若实现需要违反其中规则，必须先修改并评审设计文档，不能在代码中静默绕过。

## 2. 基础技术约束

- Java 21、Spring Boot、Maven；
- Vue 3、TypeScript、Vite；
- JUnit、Spring Boot Test、Vitest、Vue Test Utils、Playwright；
- 生产交付保持一个可执行 JAR 和一个 Docker 镜像；
- 公开运行时代码只能读取审核后的 `backend/src/main/resources/public-data/`；
- 不得读取私有 Obsidian 知识库、候选快照、原始日报、未审核截图或凭据。

## 3. Java 禁止项

生产代码和测试代码都禁止：

- 使用 `var`；
- 声明 `record`；
- 引入 Lombok；
- 使用 Lombok 的 `@Data`、`@Getter`、`@Setter`、`@Value`、`@Builder` 等注解；
- 使用字段注入；
- 通过包含大量依赖或业务方法的 Controller 基类复用代码；
- Controller 直接访问 Repository 实现或基础设施类；
- API 直接返回基础设施对象或私有源对象。

局部变量必须写出明确类型，构造器依赖必须使用显式类型和构造器注入。

## 4. 普通不可变类

用于 API DTO、领域模型、快照模型和回答结果的值对象默认采用普通不可变类：

```java
public final class ExampleValue {
    private final String id;
    private final List<String> items;

    public ExampleValue(String id, List<String> items) {
        this.id = id;
        this.items = List.copyOf(items);
    }

    public String getId() {
        return id;
    }

    public List<String> getItems() {
        return items;
    }
}
```

必须遵守：

- 类使用 `final`，除非设计明确要求继承；
- 字段使用 `private final`；
- 构造器显式声明全部必要参数；
- getter 使用 `getXxx()`，布尔值使用 `isXxx()`；
- 不提供 setter；
- 集合和 Map 在构造器中防御性复制；
- 需要值语义时显式实现 `equals()`、`hashCode()` 和 `toString()`；
- Jackson 反序列化使用显式构造器配置，不依赖编译器保留的隐式参数名；
- 构造器不执行网络、文件或数据库访问。

## 5. 后端模块与分层

后端使用单 Maven 模块，Java 包按业务模块组织：

```text
com.portfolio.agent
├─ common
├─ portfolio
│  ├─ controller
│  ├─ dto
│  ├─ service
│  ├─ domain
│  ├─ repository
│  ├─ mapper
│  ├─ validation
│  └─ exception
└─ answer
   ├─ controller
   ├─ dto
   ├─ service
   ├─ domain
   ├─ engine
   ├─ gateway
   ├─ adapter
   ├─ mapper
   └─ exception
```

只创建当前功能实际需要的包，不为未来能力预建空目录。

### 5.1 `controller`

- 处理 HTTP 协议、输入校验和 DTO 转换；
- 不编写核心业务规则；
- 不直接读取 JSON、文件或 Repository 实现；
- Controller 方法保持短小，一个方法对应一个明确用例。

### 5.2 `service`

- 编排一个完整用例；
- 依赖所属模块的领域模型、查询结果和端口接口；
- 负责事务或流程边界；
- 不依赖具体 Web Controller；
- Portfolio Service 不得依赖 Controller 或 Response DTO。

### 5.3 `domain`

- 保存所属模块的业务模型、枚举和业务规则；
- 不依赖 Controller、Repository 实现或具体外部适配器；
- 项目状态和贡献类型是权威事实，不得在 DTO 层重新推断。

### 5.4 `repository`、`validation` 与 `mapper`

- `portfolio.repository` 保存公开作品集仓储接口；
- `portfolio.repository.file` 保存当前 JSON 文件仓储实现；
- `portfolio.validation` 校验公开快照的引用、状态和公开边界；
- `portfolio.mapper` 与 `answer.mapper` 只负责将 Service/Domain 结果映射为 Response DTO；
- Repository 对象和 Domain 对象不得直接作为 API 响应。

### 5.5 `engine`、`gateway` 与 `adapter`

- `answer.engine` 只接收 Answer 自有知识模型，不读取 Repository；
- `answer.gateway` 定义 Answer 所需的跨模块能力，不表达 URL、HTTP Method、Feign 或文件结构；
- `answer.adapter.portfolio` 是当前唯一允许同时理解 Portfolio 与 Answer 类型的位置；
- 本地适配器负责从一个 Portfolio 快照筛选公开事实，并转换为 Answer 自有类型；
- 后续外部模型、向量服务或远程 Portfolio 调用只能通过 Adapter 接入，Service 与 Domain 不得直接依赖厂商客户端。

### 5.6 `common`

- 只保存至少被两个业务模块需要的机制；
- 不保存作品集或问答的具体业务规则；
- 禁止将无法归类的代码默认放入 `common`。

## 6. 依赖规则

- `portfolio.controller → portfolio.service → portfolio.domain/repository`；
- `portfolio.repository.file → portfolio.repository/domain/validation`；
- `portfolio.controller → portfolio.mapper → portfolio.dto.response`；
- `answer.controller → answer.service → answer.domain/engine/gateway`；
- `answer.engine.deterministic → answer.engine/domain`；
- `answer.adapter.portfolio → answer.gateway/domain + portfolio.repository/domain`；
- Answer Service、Domain、Engine 和 Gateway 不得 import Portfolio 包；
- 只有 `answer.adapter.portfolio` 可以把 Portfolio 类型翻译为 Answer 类型；
- Portfolio Service 不得依赖 Controller 或 Response DTO；
- `portfolio` 不得依赖 `answer`；
- `common` 不得依赖任何具体业务模块；
- 禁止循环依赖；
- 跨模块调用必须通过 Gateway，不得引用对方 Controller 或 Response DTO；
- 模块化单体内部禁止 Feign、Spring HTTP 客户端和 localhost 远程自调用；
- 当前调用必须通过 Java Gateway 接口在同一 Spring Boot 进程内完成。

## 7. 异常与错误响应

- 公共异常机制位于 `common.exception`；
- 具体业务异常位于所属模块；
- 公共错误码只保存跨模块通用错误，业务错误码必须由所属模块定义；
- 禁止建立包含所有业务模块错误的巨型公共枚举；
- 启动期公开快照数据异常位于文件仓储实现包 `portfolio.repository.file`，当前由 `InvalidPortfolioSnapshotException` 表达；
- API 统一由 `GlobalExceptionHandler` 转换错误响应；
- 未知异常返回固定公开文案；
- 不得向前端返回堆栈、内部路径、主机地址、凭据或原始异常消息；
- 不使用 Controller 继承实现全局异常处理；
- 错误码一经公开不得随意改变语义。

## 8. 前端目录与依赖

前端使用轻量 feature-first 结构：

```text
src/
├─ app
├─ pages
├─ features
│  ├─ portfolio
│  └─ agent
├─ shared
└─ test
```

依赖方向为：

```text
app/pages → features → shared
```

约束如下：

- 页面负责路由级编排；
- 功能模块保存自己的 API、类型和组件；
- `shared` 只保存真实复用内容；
- 测试文件与实现文件就近放置；
- 不允许 feature 反向依赖 page；
- 未经设计确认不得新增全局状态库或通用请求框架；
- 样式调整必须单独评审并运行视觉及响应式验收。

## 9. 命名和代码规模

- 类名使用大驼峰，方法、参数和变量使用小驼峰；
- 名称必须表达业务含义，避免 `data`、`info`、`util` 等过度宽泛名称；
- 常量使用大写下划线；
- 枚举以清晰的业务名命名，不强制无意义后缀；
- 一个类只承担一个主要职责；
- 方法出现多个独立阶段时应提取命名明确的私有方法；
- 注释解释业务原因、约束或取舍，不复述代码表面行为；
- 新增公共工具类前，先确认其确实被多个模块复用。

## 10. 安全与隐私

- 公开 API 只返回人工审核的 DTO；
- 只有 `publicStatus = APPROVED` 的 Evidence 可以公开；
- 访客问题不得在 V0 中持久化或写入日志；
- 不记录私有知识库路径、内部主机、IP、Token、账号或原始证据；
- 不把提示词当作安全边界；
- 构建和发布前必须运行隐私扫描；
- 新增外部模型时，模型只能接收公开快照中的审核内容。

## 11. 测试要求

- 新功能和缺陷修复遵循 TDD：RED、GREEN、REFACTOR；
- 纯移动或重命名前先取得基线测试结果，迁移后运行对应测试；
- API 改动必须有契约测试；
- 不可变类必须验证集合防御性复制和值语义；
- 前端功能必须覆盖成功、加载、空状态和错误状态；
- 路由和静态资源交付必须有端到端验证；
- 声明完成前必须运行新鲜的全量发布验证。

## 12. 自动质量门禁

`scripts/code-quality-check.ps1` 必须检查生产与测试 Java 代码，阻断：

- `var`；
- `record`；
- Lombok import；
- Lombok 数据类注解。

`scripts/architecture-check.ps1` 必须独立检查生产与测试 Java 代码，至少阻断：

- `common` 依赖 Portfolio 或 Answer；
- Portfolio Service 依赖 Controller 或 DTO；
- Answer 核心层依赖 Portfolio；
- `answer.adapter.portfolio` 之外的 Answer 代码依赖 Portfolio；
- Answer Engine 依赖 Answer 的 Service、Controller、DTO、Gateway、Adapter、Exception 或 Repository；
- Portfolio/Answer Controller 依赖 Repository、Adapter、Engine 或 Validation；
- Portfolio 依赖 Answer；
- 已移除的 `api/application/infrastructure/domain.model/domain.repository` 包重新出现。

依赖规则必须同时检查 import 与代码体中的 `com.portfolio.agent...` 全限定引用。检查前先处理 Java Unicode escape，并清除注释、字符串、字符字面量和 text block；package/import 已处理区域不得重复作为全限定引用报告。代码体全限定引用的违规输出为 `rule:file:line:normalized-reference`。

架构检查当前是独立后端门禁；接入完整发布流水线明确延期，未运行该检查时不得声称模块边界已经过验证。

`scripts/verify-release.ps1` 必须依次覆盖或调用：

1. 代码约束检查；
2. 后端测试；
3. 前端测试；
4. 前端生产构建；
5. 隐私检查；
6. Maven 打包；
7. Playwright 端到端验收；
8. 最终制品启动与 HTTP 检查。

任何一项失败都不能声明发布验证通过。

## 13. 变更规则

- 新功能先更新权威需求或设计，再进入实施计划；
- 不因未来可能需要而提前创建空模块、空接口或空目录；
- 不修改用户已有 Git 暂存内容；
- 未经明确授权不 stage、commit 或 push；
- 参考项目 `D:\code\d11_server` 只读，不复制其业务代码或内部敏感信息；
- 本规范与产品安全边界冲突时，以更严格的安全边界为准。
