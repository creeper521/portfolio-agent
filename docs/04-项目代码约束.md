# 实习作品集 Agent：项目代码约束

> 适用项目：`D:\code\agent`  
> 生效日期：2026-07-16
> 适用范围：生产代码、测试代码、构建脚本和后续功能开发

## 1. 目的

本规范用于保证项目在继续扩展作品集、问答引擎和后续模型能力时，仍保持清晰的业务边界、显式的数据结构、统一的前端设计系统、稳定的错误契约和可验证的安全边界。

本规范是代码审查和发布验证的依据。若实现需要违反其中规则，必须先修改并评审设计文档，不能在代码中静默绕过。

## 2. 基础技术约束

- Java 21、Spring Boot、Maven；
- Vue 3、TypeScript、Vite；
- JUnit、Spring Boot Test、Vitest、Vue Test Utils、Playwright；
- 生产交付保持一个可执行 JAR 和一个 Docker 镜像；
- 公开运行时代码只能读取审核后的 `backend/src/main/resources/public-data/`；
- 不得读取私有 Obsidian 知识库、候选快照、原始日报、未审核截图或凭据。

## 3. Java 禁止项

生产代码和测试代码都禁止：

- 使用 `var`；
- 声明 `record`；
- 引入 Lombok；
- 使用 Lombok 的 `@Data`、`@Getter`、`@Setter`、`@Value`、`@Builder` 等注解；
- 使用字段注入；
- 通过包含大量依赖或业务方法的 Controller 基类复用代码；
- Controller 直接访问 Repository 实现或基础设施类；
- API 直接返回基础设施对象或私有源对象。

局部变量必须写出明确类型，构造器依赖必须使用显式类型和构造器注入。

## 4. 普通不可变类

用于 API DTO、领域模型、快照模型和回答结果的值对象默认采用普通不可变类：

```java
public final class ExampleValue {
    private final String id;
    private final List<String> items;

    public ExampleValue(String id, List<String> items) {
        this.id = id;
        this.items = List.copyOf(items);
    }

    public String getId() {
        return id;
    }

    public List<String> getItems() {
        return items;
    }
}
```

必须遵守：

- 类使用 `final`，除非设计明确要求继承；
- 字段使用 `private final`；
- 构造器显式声明全部必要参数；
- getter 使用 `getXxx()`，布尔值使用 `isXxx()`；
- 不提供 setter；
- 集合和 Map 在构造器中防御性复制；
- 需要值语义时显式实现 `equals()`、`hashCode()` 和 `toString()`；
- Jackson 反序列化使用显式构造器配置，不依赖编译器保留的隐式参数名；
- 构造器不执行网络、文件或数据库访问。

## 5. 后端模块与分层

后端使用单 Maven 模块，Java 包按业务模块组织：

```text
com.portfolio.agent
├─ common
├─ portfolio
│  ├─ controller
│  ├─ dto
│  ├─ service
│  │  └─ result
│  ├─ domain
│  ├─ repository
│  ├─ mapper
│  ├─ validation
│  └─ exception
└─ answer
   ├─ controller
   ├─ dto
   ├─ service
   ├─ domain
   ├─ engine
   ├─ gateway
   ├─ adapter
   ├─ mapper
   └─ exception
```

只创建当前功能实际需要的包，不为未来能力预建空目录。

### 5.1 `controller`

- 处理 HTTP 协议、输入校验和 DTO 转换；
- 不编写核心业务规则；
- 不直接读取 JSON、文件或 Repository 实现；
- Controller 方法保持短小，一个方法对应一个明确用例。

### 5.2 `service`

- 编排一个完整用例；
- 依赖所属模块的领域模型、查询结果和端口接口；
- 负责事务或流程边界；
- 不依赖具体 Web Controller；
- Portfolio Service 不得依赖 Controller 或 Response DTO。

### 5.3 `domain`

- 保存所属模块的业务模型、枚举和业务规则；
- 不依赖 Controller、Repository 实现或具体外部适配器；
- 项目状态和贡献类型是权威事实，不得在 DTO 层重新推断。

### 5.4 `repository`、`validation` 与 `mapper`

- `portfolio.repository` 保存公开作品集仓储接口；
- `portfolio.repository.file` 保存当前 JSON 文件仓储实现；
- `portfolio.validation` 校验公开快照的引用、状态和公开边界；
- `portfolio.mapper` 与 `answer.mapper` 只负责将 Service/Domain 结果映射为 Response DTO；
- Repository 对象和 Domain 对象不得直接作为 API 响应。

### 5.5 `engine`、`gateway` 与 `adapter`

- `answer.engine` 只接收 Answer 自有知识模型，不读取 Repository；
- `answer.gateway` 定义 Answer 所需的跨模块能力，不表达 URL、HTTP Method、Feign 或文件结构；
- `answer.adapter.portfolio` 是当前唯一允许同时理解 Portfolio 与 Answer 类型的位置；
- 本地适配器负责从一个 Portfolio 快照筛选公开事实，并转换为 Answer 自有类型；
- 后续外部模型、向量服务或远程 Portfolio 调用只能通过 Adapter 接入，Service 与 Domain 不得直接依赖厂商客户端。

### 5.6 `common`

- 只保存至少被两个业务模块需要的机制；
- 不保存作品集或问答的具体业务规则；
- 禁止将无法归类的代码默认放入 `common`。

## 6. 依赖规则

- `portfolio.controller → portfolio.service → portfolio.domain/repository`；
- `portfolio.repository.file → portfolio.repository/domain/validation/exception`；
- `portfolio.validation → portfolio.domain/exception`；
- `portfolio.controller → portfolio.mapper → portfolio.dto.response`；
- `answer.controller → answer.service → answer.domain/engine/gateway`；
- `answer.engine.deterministic → answer.engine/domain`；
- `answer.adapter.portfolio → answer.gateway/domain + portfolio.repository/domain`；
- Answer Service、Domain、Engine 和 Gateway 不得 import Portfolio 包；
- 只有 `answer.adapter.portfolio` 可以把 Portfolio 类型翻译为 Answer 类型；
- Portfolio Service 不得依赖 Controller 或 Response DTO；
- `portfolio` 不得依赖 `answer`；
- `common` 不得依赖任何具体业务模块；
- 禁止循环依赖；
- 跨模块调用必须通过 Gateway，不得引用对方 Controller 或 Response DTO；
- 模块化单体内部禁止 Feign、Spring HTTP 客户端和 localhost 远程自调用；
- 当前调用必须通过 Java Gateway 接口在同一 Spring Boot 进程内完成。

## 7. 异常与错误响应

- 公共异常机制位于 `common.exception`；
- 具体业务异常位于所属模块；
- 公共错误码只保存跨模块通用错误，业务错误码必须由所属模块定义；
- 禁止建立包含所有业务模块错误的巨型公共枚举；
- 启动期公开快照数据异常位于模块异常包 `portfolio.exception`，当前由 `InvalidPortfolioSnapshotException` 表达；
- API 统一由 `GlobalExceptionHandler` 转换错误响应；
- 未知异常返回固定公开文案；
- 不得向前端返回堆栈、内部路径、主机地址、凭据或原始异常消息；
- 不使用 Controller 继承实现全局异常处理；
- 错误码一经公开不得随意改变语义。

## 8. 前端目录与依赖

前端使用轻量 feature-first 结构：

```text
src/
├─ app
│  ├─ router.ts
│  └─ styles
├─ pages
├─ features
│  ├─ portfolio
│  ├─ public-content
│  ├─ audience
│  ├─ home
│  ├─ projects
│  ├─ timeline
│  ├─ evidence
│  └─ agent
├─ shared
└─ test
```

依赖方向为：

```text
app/pages → features → shared
```

约束如下：

- 页面负责路由级编排；
- 功能模块保存自己的 API、类型和组件；
- `shared` 只保存真实复用内容；
- 页面不得直接散布临时 API 映射；公开内容统一经过 `PublicContentRepository` 或等价的明确适配边界；
- 后端重构期间允许使用 `previewPublicContentRepository`，但其中只能复制或整理已批准的公开事实，不得加入演示性假项目、假数字或假证据；
- 后续切换真实 API 时替换 Repository 实现，不得为联调推翻页面组件和路由结构；
- 测试文件与实现文件就近放置；
- 不允许 feature 反向依赖 page；
- 未经设计确认不得新增全局状态库或通用请求框架；
- 样式调整必须单独评审并运行视觉及响应式验收。

### 8.1 固定路由与页面职责

正式前端路由固定为：

```text
/                 首页四层体验
/projects         项目案卷目录
/projects/:slug   项目详情
/timeline         公开成长时间线
/evidence         证据中心
/agent            完整 Agent 工作台
```

- 首页只包含 Hero、可信度摘要、身份选择与轻对话、探索入口四层；
- 项目、时间线和证据的完整内容必须放在各自路由，不得重新堆回首页；
- `/agent` 承载本地会话、完整对话和证据工作台；
- 404 页面必须明确表达“未公开或地址失效”，不得暴露内部标识或路径。

### 8.2 前端设计权威与色彩禁区

前端唯一视觉基线为：

```text
C:\Users\WIN10\Documents\杂项\实习学习-Obsidian\.superpowers\brainstorm\homepage-flow-20260715\content\portfolio-home-prototype-v1.html
```

设计实施规范为：

```text
docs/superpowers/specs/2026-07-16-portfolio-frontend-full-rebuild-design.md
```

发生冲突时，先遵循已审批设计文档；需要改变方向时先更新设计，不得在组件中自行发明第二套主题。

必须遵守：

- 正式色盘只使用纸张色、墨色、灰褐色、规则线和暗红；
- 禁止墨绿色、青绿色、绿色状态色、紫色、蓝绿色、装饰性渐变和霓虹光晕；
- 禁止恢复旧前端的青绿色主题、圆角卡片、胶囊堆叠、玻璃拟态和通用 SaaS 面板；
- 成功、通过、同步和可信状态通过明确文字、墨色/灰褐色、暗红细节和符号共同表达，不使用绿色；
- 不允许旧视觉组件以“临时兼容”或“次级页面复用”为理由重新进入正式路由；
- 组件按业务语义命名，禁止建立掩盖业务差异的万能 `Card` 组件；
- 缺少姓名和联系方式时直接隐藏；缺少内容时使用“待公开 / 资料准备中”，不得显示 `[待填写]`。

### 8.3 Agent 与本地状态

- 本地会话键使用版本化命名，当前为 `portfolio.agent.sessions.v1`；
- 会话只保存在当前浏览器，默认 7 天过期，支持单条删除和全部清除；
- 后端不得持久化访客会话或问题，也不得将访客问题写入应用日志；
- 面板宽度使用 `portfolio.workspace.split.v1` 保存；
- 会话栏边界为 `220–380px`，证据栏边界为 `380–760px`，对话栏最低 `600px`；
- `>=1280px` 使用完整三栏，`1220–1279px` 使用压缩三栏，`<1220px` 证据转抽屉，`<=980px` 会话也转抽屉；
- 分隔条必须同时支持 Pointer Events、双击复位、方向键、Shift 加速和 Home 复位；
- 抽屉必须提供 `aria-expanded`、Escape 关闭、遮罩关闭和焦点恢复。

### 8.4 响应式、动效与无障碍

- 页面不得出现横向滚动；
- 正文阅读宽度不得随超宽屏无限增长；
- 响应式优先重排内容和转抽屉，不得只靠缩小字体；
- 所有主要交互使用真实按钮或链接，不使用可点击 `div`；
- 键盘必须能够完成导航、身份选择、推荐问题、证据选择和面板调整；
- 必须提供清晰的 `:focus-visible`；
- `prefers-reduced-motion: reduce` 下关闭非必要位移、错峰和逐字动效；
- Agent 新回答不得逐字触发读屏器重复播报；
- 正式页面不得因追求编辑感而把核心中文正文缩小到难以阅读。

## 9. 命名和代码规模

- 类名使用大驼峰，方法、参数和变量使用小驼峰；
- 名称必须表达业务含义，避免 `data`、`info`、`util` 等过度宽泛名称；
- 常量使用大写下划线；
- 枚举以清晰的业务名命名，不强制无意义后缀；
- 一个类只承担一个主要职责；
- 方法出现多个独立阶段时应提取命名明确的私有方法；
- 注释解释业务原因、约束或取舍，不复述代码表面行为；
- 新增公共工具类前，先确认其确实被多个模块复用。

## 10. 安全与隐私

- 公开 API 只返回人工审核的 DTO；
- 只有 `publicStatus = APPROVED` 的 Evidence 可以公开；
- 访客问题不得在后端持久化或写入日志；浏览器本地会话按第 8.3 节保存；
- 不记录私有知识库路径、内部主机、IP、Token、账号或原始证据；
- 不把提示词当作安全边界；
- 构建和发布前必须运行隐私扫描；
- 新增外部模型时，模型只能接收公开快照中的审核内容。

## 11. 测试要求

- 新功能和缺陷修复遵循 TDD：RED、GREEN、REFACTOR；
- 纯移动或重命名前先取得基线测试结果，迁移后运行对应测试；
- API 改动必须有契约测试；
- 不可变类必须验证集合防御性复制和值语义；
- 前端功能必须覆盖成功、加载、空状态和错误状态；
- 路由和静态资源交付必须有端到端验证；
- 前端独立开发阶段的 Playwright 必须由 Vite 启动，不得启动 Java；它只能 mock `GET /api/v1/public-content` 与 `POST /api/v1/answers`，不得拦截静态资源、导航或其他 API；
- 前端视觉验收至少覆盖 `2048×1080`、`1440×900`、`1219×900`、`980×800` 和 `390×844`；
- 视觉验收必须检查无横向溢出、无绿色/旧主题残留、Agent 对话左落位、抽屉切换和控制台错误；
- 声明完成前必须运行新鲜的全量发布验证。

## 12. 自动质量门禁

`scripts/code-quality-check.ps1` 必须检查生产与测试 Java 代码，阻断：

- `var`；
- `record`；
- Lombok import；
- Lombok 数据类注解。

`scripts/architecture-check.ps1` 必须独立检查生产与测试 Java 代码，至少阻断：

- `common` 依赖 Portfolio 或 Answer；
- Portfolio Service 依赖 Controller 或 DTO；
- Answer 核心层依赖 Portfolio；
- `answer.adapter.portfolio` 之外的 Answer 代码依赖 Portfolio；
- Answer Engine 依赖 `answer.engine`、`answer.domain` 之外的任何项目内部包；
- Portfolio/Answer Controller 依赖 Repository、Adapter、Engine 或 Validation；
- Portfolio 依赖 Answer；
- Java 声明的 package 与 Maven source-set 内的文件目录不一致；
- 已移除的 `api/application/infrastructure/domain.model/domain.repository` 包重新出现。

依赖规则必须以源码声明的 normalized package 判断来源模块与层，不得使用文件路径代替 package 分类；同时，Maven `main/java` 与 `test/java` 下的文件目录必须与 package path 一致，默认 package 也会被拒绝。依赖规则同时检查 import 与代码体中的 `com.portfolio.agent...` 全限定引用。检查前先处理 Java Unicode escape，并清除注释、字符串、字符字面量和 text block；package/import 已处理区域不得重复作为全限定引用报告。代码体全限定引用的违规输出为 `rule:file:line:normalized-reference`，目录不一致输出 `package-path-mismatch:file:line:package ...;`。

`answer-engine-boundary` 对项目内部引用采用白名单：`answer.engine` 只允许依赖 `answer.engine` 与 `answer.domain`；JDK、Spring 注解等非项目内部引用不受该规则限制。

架构检查同时作为独立后端门禁和完整发布流水线的一部分；未运行该检查时不得声称模块边界已经过验证。

`scripts/verify-release.ps1` 必须依次覆盖或调用：

1. 代码约束检查器自测与源码扫描；
2. 架构检查器自测与源码扫描；
3. 静态 bundle 检查器自测；
4. 可选的前端依赖安装；
5. 前端测试与生产构建；
6. Maven `clean package`，其中包含后端测试；
7. 隐私检查器自测，以及公开快照和前端构建产物扫描；
8. JAR 清单与禁止项检查、解包、静态资源 ordinal 大小写敏感路径集合和逐文件 SHA-256 对比、打包内容隐私扫描；
9. 启动最终 JAR，确认新进程拥有监听端口，解析聚合 API JSON 关键字段，再运行无 mock 的 Playwright 联调；
10. Dockerfile 检查（显式跳过时除外）。

任何一项失败都不能声明发布验证通过。

Playwright 拓扑必须明确分离：

```powershell
# Frontend-only visual/interaction acceptance with API mocks
npm.cmd --prefix frontend run test:e2e

# Full packaged-JAR frontend/backend integration
powershell -ExecutionPolicy Bypass -File scripts/run-jar-e2e.ps1
```

前端独立命令必须由 Vite 启动页面，只 mock 两个明确的公开 API，且不得启动 Java。完整联调命令不得注册 API mock；运行前必须完成一次新的 `npm.cmd --prefix frontend run build` 和 `mvn.cmd -f backend/pom.xml clean package`，再由脚本启动 JAR、等待真实聚合 API 就绪、运行浏览器用例并在 `finally` 中恢复原环境变量和清理进程。Playwright 非零退出码必须在清理完成后原样返回；启动或 readiness 异常也必须返回非零。

## 13. 变更规则

- 新功能先更新权威需求或设计，再进入实施计划；
- 不因未来可能需要而提前创建空模块、空接口或空目录；
- 不修改用户已有 Git 暂存内容；
- 未经明确授权不 stage、commit 或 push；
- 删除或替换页面前先确认无剩余 import；全量重构后禁止回引已经废弃的旧视觉组件；
- 参考项目 `D:\code\d11_server` 只读，不复制其业务代码或内部敏感信息；
- 本规范与产品安全边界冲突时，以更严格的安全边界为准。
