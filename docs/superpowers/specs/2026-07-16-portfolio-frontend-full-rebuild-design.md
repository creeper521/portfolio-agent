## 1. 设计结论

正式前端不再延续现有页面的任何视觉语言、页面构图和交互表现，而是以目标原型为唯一的前端设计基线，进行基于 Vue Router 的多页面全量重构。

这里的“全量参考”包括目标原型的色彩体系、字体关系、排版比例、暗色对话区、纸张质感、规则线、编号系统、内容密度、动效节奏、响应式逻辑和页面气质。后续页面可以根据内容扩展，但不能重新回到旧前端的设计语言。

重构不是把原型 HTML 原样塞入 Vue，也不是给现有页面换一层皮肤，而是：

1. 技术层只保留已经可靠的 Vue、TypeScript、Vite、API 类型、错误处理和测试基础；
2. 设计层全量放弃现有页面结构、现有组件外观、青绿色主题、圆角卡片和通用 SaaS 面板语言；
3. 将原型拆解为可维护的路由、页面、领域组件、状态和设计令牌；
4. 以公开后端数据为事实来源，不虚构履历、项目、指标、证据或联系方式；
5. 在数据尚未开放时仍提供完整页面结构，但使用诚实、克制的待公开状态；
6. 保持作品集、证据和 Agent 三者之间可追溯，而不是把它们做成互不关联的栏目。

“保留技术基础”不代表保留旧页面的 DOM、CSS、组件造型或布局。任何旧组件只有在去除原有视觉、重新按目标原型设计后，才可以继续承担业务职责。

## 2. 产品定位

产品不是普通个人主页，也不是通用聊天机器人。它是一份可探索、可追问、可核验的“工程成长档案”。

访客首先建立对作者定位和可信度的印象，然后根据自己的身份选择问题，进一步查看项目、成长时间线和证据；需要连续追问时，再进入完整 Agent 工作台。

核心体验链路为：

```text
强视觉首页
  -> 选择访客身份
  -> 浏览身份化推荐问题
  -> 获得带证据引用的摘要回答
  -> 进入项目 / 时间线 / 证据
  -> 或携带上下文进入完整 Agent
  -> 在对话与证据之间持续核验
```

## 3. 目标与非目标

### 3.1 目标

- 第一屏形成鲜明、克制且可记忆的视觉印象。
- 首页只承担定位、可信度、轻问答和分流，不堆放工程细节。
- 让项目、成长过程、证据和 Agent 形成统一的信息网络。
- 对不同访客提供不同的内容入口和表达侧重，但不改变事实。
- 即使公开数据有限，也不通过虚构内容填满页面。
- 支持桌面、平板和手机上的完整核心流程。
- 为后续动态公开内容、RAG 和更多证据类型预留稳定边界。

### 3.2 非目标

- 不建设通用 AI 聊天产品。
- 不在前端引入登录、云端会话、多用户协作或管理后台。
- 不让内容配置注入任意 HTML、CSS、JavaScript 或 Prompt。
- 不为了视觉完整而虚构项目数量、经历年限、成果数据或联系方式。
- 不在首页复刻项目详情、完整时间线或证据中心。
- 不要求第一阶段实现 SSE 或 WebSocket 流式响应；可以对完整响应做前端逐字呈现。

## 4. 已确认的产品决策

1. 重构范围为完整目标版前端，而非只完成首页或 V0。
2. 现有正式前端的视觉、页面结构、组件外观和交互表现全部废弃，不作为新设计约束，也不允许在新页面中局部回用。
3. 采用 Vue Router 多路由架构。
4. 正式路由为：
   - `/`
   - `/projects`
   - `/projects/:slug`
   - `/timeline`
   - `/evidence`
   - `/agent`
5. 缺失事实使用“待公开”“资料准备中”等显式状态，不使用假数据。
6. 缺失的姓名、联系方式等字段直接隐藏，不展示 `[待填写]`。
7. 会话只保存在当前浏览器，默认 7 天过期，支持手动清除。
8. 首页轻对话与完整 Agent 共享身份、问题和公开证据语义。
9. 首页只给出摘要回答；连续追问和完整能力进入 `/agent`。

## 5. 信息架构与全局导航

### 5.1 全局导航

桌面导航固定在顶部，保持薄、暗、克制。建议顺序：

```text
概览 / 项目 / 时间线 / 证据 / 完整 Agent
```

左侧为作者标识和 `ENGINEERING DOSSIER` 类型说明；右侧为主导航。当前路由通过暗红下划线、文字色和小型序号共同标识，不使用大面积胶囊按钮。

移动端不强行保留完整横向导航。顶部保留作者标识和菜单触发器，展开后使用整幅纸张式目录层。导航打开时锁定背景滚动，支持 Escape 关闭并恢复焦点。

### 5.2 页面间上下文

以下上下文允许跨页面携带：

- 当前访客身份；
- 首页已选择的问题；
- 当前项目；
- 当前证据引用；
- 从首页进入 Agent 时的首个问题和摘要上下文。

适合分享和刷新后恢复的状态放入 URL 查询参数，例如：

```text
/agent?role=interviewer&question=...
/evidence?project=sql-audit&evidence=E-01
/timeline?project=sql-audit
```

浏览器会话内容、面板宽度等个人状态不写入 URL。

## 6. 页面设计

### 6.1 首页 `/`

首页严格保留四层内容。

#### 第一层：强视觉 Hero

目标是让访客在数秒内记住作者定位，而不是立即阅读项目细节。

内容结构：

- 档案编号、卷宗标签或当前公开版本；
- 作者姓名；姓名不存在时隐藏姓名行并提升定位为主标题；
- 一句清晰定位；
- 一句核心主张；
- 向下探索提示；
- 克制的工程坐标、细线、编号和文字动效。

Hero 不展示项目卡片、技术栈列表、长篇自我介绍和证据清单。它使用大字号衬线体、紧凑的无衬线说明和等宽编号形成编辑出版物感。

首屏高度在桌面端约为 `680–780px`，但要兼顾低高度屏幕，不强制制造首屏裁切。动效围绕文字揭示、线条展开、编号定位和细微视差，不使用粒子背景、霓虹光晕或大面积渐变。

#### 第二层：极简可信度摘要

摘要只建立可信感，不在首页解释所有细节。

每个统计项包含：

- 后端计算或公开数据确认的数值；
- 简短标签；
- 指向对应页面的完整点击区域；
- 可选的辅助说明。

可能的数据包括公开时间跨度、核心项目数、已批准证据数。实际没有数据时，该统计项不出现，不用 `0` 或假数字填充。统计项数量允许动态为 1–4 个，布局仍需成立。

#### 第三层：身份选择与首页轻对话

四种身份为：

- 技术面试官 `INTERVIEWER`
- 未来导师 `MENTOR`
- HR / 招聘者 `HR`
- 普通访客 `GUEST`

身份使用四宫格或在窄屏下改为两列/单列目录。每个身份项包括编号、中文名称和一句目的说明，不使用头像、插画或彩色图标。

选择身份后：

1. 视觉上明确当前身份；
2. 右侧推荐问题即时切换；
3. 问题仍来自同一公开事实，只改变排序、内容侧重和表达方式；
4. 点击问题后在首页原位显示一段摘要回答；
5. 回答附带最多三条可点击证据或项目引用；
6. 回答底部提供“继续追问”与“查看相关材料”入口；
7. “继续追问”携带身份和问题进入 `/agent`。

首页轻对话最多承载 2–3 轮。达到上限、问题过于复杂或访客主动继续时进入完整 Agent。首页不出现会话列表、模型设置、长输入历史或复杂工具栏。

#### 第四层：探索入口

固定提供四个入口：

- 项目
- 时间线
- 证据
- 完整对话

入口采用大型编辑目录或索引块，而非同质化圆角卡片。每个入口包含编号、标题、一句价值说明和方向提示。整个区域是可点击目标，并具有清晰的键盘焦点状态。

### 6.2 项目目录 `/projects`

项目目录是一份“工程案卷目录”，不是营销卡片瀑布流。

页面结构：

1. 页面标题与公开内容说明；
2. 可选的主题/状态筛选；
3. 项目索引列表；
4. 数据不足时的资料准备状态。

每个项目条目显示：

- 档案编号；
- 项目名称；
- 一句话问题定义；
- 角色与职责；
- 状态；
- 时间范围；
- 已公开证据数；
- 精简技术标签。

项目数量较少时使用纵向案卷索引，避免为了“看起来丰富”复制卡片。筛选项不足两个时不显示筛选控件。

### 6.3 项目详情 `/projects/:slug`

项目详情回答三个问题：为什么做、如何做、怎样证明。

推荐内容顺序：

1. 项目封面与状态；
2. 问题背景与约束；
3. 个人职责；
4. 方案选择与关键决策；
5. 实现路径；
6. 验证与结果；
7. 复盘与下一步；
8. 关联证据；
9. 关联时间线；
10. 针对该项目继续询问 Agent。

详情采用正文阅读列与右侧档案边栏。边栏用于状态、日期、技术标签、证据索引和章节目录，不承载主要叙事。小屏下边栏并入正文顶部。

没有公开的章节不渲染空标题；若整个关键区仍在准备，则显示明确的准备状态和已经可以核验的部分。

### 6.4 成长时间线 `/timeline`

时间线展示能力形成过程，而不是仅列日期。

每个事件应包括：

- 日期或阶段；
- 事件名称；
- 当时的问题；
- 做出的行动或决策；
- 对能力的影响；
- 关联项目与证据。

桌面端使用非对称纵向时间轴：日期和编号形成稳定轴线，正文在右侧展开；重大节点可以跨列。移动端收敛为单列，保持顺序和关联关系。

支持按项目聚焦，但不在第一阶段提供复杂多维筛选。没有事件时展示“公开时间线正在整理”，并提供项目和 Agent 入口。

### 6.5 证据中心 `/evidence`

证据中心不是下载列表，而是用于回答“这句话凭什么可信”。

页面结构：

1. 证据中心说明及公开边界；
2. 类型、项目、时间筛选；
3. 证据目录；
4. 证据详情预览或侧栏；
5. 与项目和 Agent 的双向关联。

证据条目至少包含：

- 公开证据编号；
- 类型；
- 标题；
- 摘要；
- 时间；
- 关联项目；
- 公开状态；
- 可展示的内容或链接。

只展示 `APPROVED` 且允许公开的证据。`DRAFT`、`REJECTED`、`BLOCKED` 和私有路径不得出现在前端包、页面源码、接口响应或错误信息中。

从回答引用进入时，应自动定位并强调目标证据，但不使用强烈闪烁。若引用已失效，页面解释“该公开材料当前不可用”，同时保留返回对话的路径。

### 6.6 完整 Agent `/agent`

完整 Agent 是桌面端三栏工作台：

```text
本地会话 | Agent 对话 | 证据工作台
```

#### 左栏：本地会话

- 新建会话；
- 自动命名的会话列表；
- 身份、更新时间和可选项目上下文；
- 继续上次讨论；
- 单条删除和全部清除；
- 明确说明“只保存在当前浏览器，7 天后自动过期”。

#### 中栏：Agent 对话

- 当前问题或会话标题；
- 当前身份模式；
- 用户消息、Agent 回答、证据引用；
- 空白状态下的身份化推荐问题；
- 固定在底部的输入区；
- 发送、生成中、失败重试和停止逐字呈现状态。

对话区保持深墨色，正文阅读宽度不应随超宽屏无限增长。内容列在视觉上略偏左，给右侧证据关系留出呼吸空间；不居中悬浮成孤岛。桌面端正文最大宽度约 `820px`，根据聊天栏实际宽度动态调整左偏移，小屏恢复自然居中和满宽。

#### 右栏：证据工作台

- 当前回答引用的证据；
- 当前项目上下文；
- 证据摘要和状态；
- 在证据中心打开；
- 在回答引用之间切换。

点击回答中的证据编号，右侧工作台切换到对应条目，不打断对话滚动位置。

#### 三栏宽度

桌面默认值：

```text
会话：clamp(240px, 15vw, 320px)
对话：minmax(600px, 1fr)
证据：clamp(420px, 31vw, 720px)
```

可拖动边界：

- 会话栏 `220–380px`
- 证据栏 `380–760px`
- 对话栏最低 `600px`

两个分隔条使用 1px 可见线和约 12px 的命中区域。支持 Pointer Events、指针捕获、双击复位、方向键每次 16px、Shift + 方向键每次 48px、Home 复位，并提供正确的 `role="separator"`、方向和实时值。

宽度只在拖动结束、键盘调整或复位时写入浏览器本地存储。

#### Agent 响应式

- `>=1280px`：完整三栏；
- `1220–1279px`：压缩后的三栏；
- `<1220px`：证据栏改为右侧抽屉；
- `<=980px`：会话栏也改为抽屉，对话占满页面；
- 小屏抽屉互斥，打开后具有遮罩、焦点约束、Escape 关闭和焦点恢复。

## 7. 视觉系统

### 7.1 设计语言

关键词为：工程卷宗、编辑出版物、温暖纸张、深色墨水、暗红批注、细密索引。

本项目不再存在“旧前端主题 + 新原型主题”两套视觉系统。所有正式路由、加载状态、空状态、错误状态、抽屉、对话和证据界面，都必须属于同一套目标原型语言。

禁止把新前端做成常规 SaaS 仪表盘。除语义需要外，不使用：

- 大圆角卡片；
- 胶囊按钮堆叠；
- 玻璃拟态；
- 霓虹光晕；
- 装饰性渐变；
- 无意义头像；
- 大面积彩色图标；
- 假仪表盘数据；
- 为填空间而出现的模块。

### 7.2 核心色彩

```css
--paper: #f4eee4;
--paper-hi: #fbf7ef;
--paper-low: #e9dfd0;
--ink: #201c17;
--ink-2: #34302a;
--muted: #746b5e;
--faint: #9c9182;
--rule: #cdbfa9;
--red: #7a2e2a;
--red-hi: #b65d53;
```

目标原型源码中存在一个未实际使用的 `--green: #486153` 声明。正式前端不继承该变量，也不使用墨绿色、青绿色或绿色作为成功、可信、同步、通过、按钮或导航状态。

暗红用于当前状态、引用、关键动作和批注；通过、同步、可信等正向状态使用墨色、灰褐色、暗红细节、图形符号与明确文字共同表达。色彩不可替代文字状态。

除非用户后续专门审批新的扩展色，否则正式设计令牌不得在上述色盘外临时增加绿色、蓝绿色、青色、紫色或渐变色。

### 7.3 字体

```css
--serif: Georgia, 'Noto Serif SC', 'Songti SC', serif;
--sans: 'Noto Sans SC', 'Microsoft YaHei', sans-serif;
--mono: 'IBM Plex Mono', 'SFMono-Regular', Consolas, monospace;
```

- 姓名、页面标题、项目标题和长回答正文优先使用衬线体；
- 导航、按钮、状态和辅助说明使用无衬线体；
- 编号、日期、证据代码、标签和系统状态使用等宽体；
- 中文正文最低约 16px，重要说明和会话标题不低于 13px；
- 极小等宽文字只用于次要编号，不能承担核心信息。

### 7.4 形状、边界与阴影

- 主体采用方形或极小圆角；
- 模块通过留白、底色差和 1px 规则线分区；
- 阴影只用于抽屉和确实脱离页面层级的浮层；
- 点击区域可以大于视觉边界，但必须维持编辑式外观。

## 8. 布局与响应式

### 8.1 通用内容宽度

- 超宽屏不让正文无限拉伸；
- 主内容容器约 `1280–1440px`，Hero 和 Agent 可以突破普通容器；
- 长正文阅读列约 `680–820px`；
- 页面横向边距使用 `clamp()` 连续变化，而不是只靠几个突变断点。

### 8.2 通用断点语义

- 宽桌面：完整导航、完整多列内容；
- 中桌面/平板横屏：压缩间距和字号，减少非核心并列；
- 平板/手机：单列内容、抽屉导航、边栏并入正文；
- 小手机：避免超大标题造成逐字断裂，点击目标不小于约 44px。

### 8.3 内容重排原则

响应式不是简单缩小。重排优先级为：

1. 保留任务和信息顺序；
2. 将辅助栏转为抽屉或正文区块；
3. 移除纯装饰线和次要编号；
4. 最后才缩小字号；
5. 不允许出现水平页面滚动。

## 9. 交互与动效

### 9.1 动效原则

动效用于建立层级、解释状态和增强记忆点，不用于持续炫技。

推荐动效：

- Hero 文字与规则线分批揭示；
- 当前身份的编号、边线和推荐问题同步切换；
- 输入时使用克制的光标或标记闪动；
- 页面进入时内容轻微错峰出现；
- 证据引用定位时进行一次短暂强调；
- Agent 完整响应可以逐字呈现，但保存的是完整消息。

避免弹跳、过度缩放、连续漂浮和滚动劫持。

### 9.2 减少动态效果

当系统启用 `prefers-reduced-motion: reduce` 时：

- 关闭视差、错峰和逐字动画；
- 保留即时状态变化；
- 不因关闭动效丢失任何内容或操作能力。

### 9.3 键盘与焦点

- 所有导航、身份、问题、证据和抽屉可用键盘操作；
- `:focus-visible` 必须清晰且符合暗红/纸张视觉；
- 页面切换后焦点移动到页面主标题；
- 抽屉关闭后焦点返回触发按钮；
- 拖动分隔条具有键盘等价操作。

## 11. 数据与接口映射

### 11.1 数据真实性

所有数量、日期、职责、技术、结果和证据必须来自公开发布内容或后端 API。前端只负责展示和有限派生，不自行创造事实。

首页统计必须由后端数据计算或从可信响应字段读取。不得在组件中写死“4 个月 / 4 个项目 / 13 项证据”等示例值。

### 11.2 第一阶段接口适配

在现有后端能力有限时：

- 继续复用现有 Portfolio、Project、Answer API；
- 为新页面建立清晰的 adapter，把旧响应映射到新的前端视图模型；
- 没有数据的页面显示完整壳层和诚实空状态；
- 不把测试 fixture 或演示数据打入正式生产页面。

### 11.3 后续接口

目标接口可扩展为：

- Portfolio 首页展示模型；
- 项目目录与项目详情；
- `/api/v1/timeline`；
- `/api/v1/evidence`；
- 身份化推荐问题；
- 带 Claim 和 Evidence 引用的 Agent 回答。

接口尚未完成时，前端服务层保持替换边界，不在页面组件中散布临时判断。

## 12. 本地会话设计

浏览器会话存储建议使用版本化键，例如：

```text
portfolio.agent.sessions.v1
portfolio.agent.preferences.v1
portfolio.workspace.split.v1
```

单个会话至少保存：

- 本地 ID；
- 自动生成标题；
- 身份；
- 可选项目上下文；
- 完整用户消息和已完成的 Agent 响应；
- 创建时间和最后更新时间；
- 过期时间；
- 数据结构版本。

规则：

- 默认 7 天过期；
- 应用初始化时清理过期会话；
- 生成中的逐字状态不作为独立事实保存；
- 写入失败时不阻塞当前对话，但要提示无法保存；
- 清除全部会话需要二次确认；
- 不保存服务端未返回的证据内容；
- 不暗示本地会话已同步到云端。

## 13. 页面状态与异常处理

每个异步页面都必须具备以下状态：

### 13.1 加载

使用与最终排版一致的纸张骨架或文字状态，避免通用旋转圆圈成为页面中心。加载时保持页面标题和导航可用。

### 13.2 空状态

说明为什么当前没有内容，并提供下一步。例如：

- “公开时间线正在整理”；
- “该项目的证明材料尚未公开”；
- “选择一个身份，开始探索推荐问题”。

空状态不使用虚假示例填充。

### 13.3 错误

区分：

- 网络暂时不可用；
- 页面不存在；
- 项目或证据未公开；
- Agent 暂时无法回答；
- 本地存储失败。

错误文案避免暴露内部路径、堆栈、私有 ID 或敏感内容。可恢复错误提供重试，不可恢复错误提供返回索引的路径。

### 13.4 降级

Agent、模型或检索不可用时，仍可显示已经批准的确定性内容和相关项目/证据入口。首页、项目和证据浏览不依赖 Agent 才能成立。

## 14. 无障碍与语义

- 页面使用正确的 `header`、`nav`、`main`、`section`、`article`、`aside` 和标题层级；
- 交互元素使用按钮或链接，不用可点击 `div`；
- 身份选择使用可访问的单选语义；
- 当前路由、当前筛选和当前身份可被辅助技术感知；
- 颜色对比满足常规正文和大字号要求；
- 装饰性编号与线条不制造重复朗读；
- Agent 新回答使用克制的 live region，不逐字向读屏器重复播报；
- 证据编号的可访问名称包含证据标题，而不只读出代码。

## 15. 正式前端的取舍

### 15.1 明确放弃

- 当前首页和项目页的视觉布局；
- 当前青绿色主题；
- 当前大圆角卡片、胶囊和通用面板样式；
- 当前组件的颜色、边框、圆角、阴影、间距、字号和排列方式；
- 当前加载、空状态、错误状态和 Agent 面板的既有外观；
- 当前只围绕单一 V0 页面组织的路由结构；
- 无法承载完整目标体验的局部组件结构；
- 与目标原型冲突的间距、字号和响应式规则。

旧视觉样式不得以“临时兼容”“先复用一下”或“只在次级页面使用”为理由进入新页面。重构完成后，应删除不再被使用的旧 CSS、旧视觉组件和旧主题变量，避免两套设计长期共存。

### 15.2 只保留技术能力，不保留旧设计

- Vue 3、TypeScript、Vite 和 vue-router 基础；
- 已验证的 API 客户端、类型和错误边界；
- 可以继续成立的 Project、Evidence 和 Answer 领域模型；
- 现有测试基础设施；
- 隐私隔离、公开数据过滤和确定性回答兜底；
- 与视觉无关且行为正确的工具函数。

上述内容只属于技术和业务契约，不是设计资产。迁移前仍需逐项确认其契约是否符合新页面；一旦渲染到页面，必须使用目标原型重新设计，不能继承旧外观。

## 16. 测试与验收

### 16.1 自动化测试

- 路由和页面标题；
- 首页四层结构；
- 身份切换和推荐问题；
- 首页摘要回答及进入 Agent 的上下文；
- 项目目录和详情映射；
- 时间线与证据的空状态和正常状态；
- 只展示允许公开的证据；
- 本地会话创建、恢复、过期和清除；
- Agent 工作台宽度约束、键盘调整和复位；
- API 错误和降级；
- `prefers-reduced-motion` 行为。

### 16.2 浏览器端到端测试

至少覆盖：

1. 首页选择技术面试官并打开推荐问题；
2. 从摘要回答进入完整 Agent；
3. 从回答引用打开证据；
4. 从项目详情进入关联时间线；
5. 新建本地会话、刷新恢复、清除；
6. 桌面端拖动两个 Agent 分隔条；
7. 1219px 下证据抽屉；
8. 980px 及以下的双抽屉；
9. 键盘完成主要导航和身份选择；
10. 404、接口失败和无公开数据状态。

### 16.3 视觉验收

重点视口：

- `2048 × 1080`
- `1440 × 900`
- `1280 × 800`
- `1219 × 900`
- `980 × 800`
- `390 × 844`

验收要求：

- 无横向页面溢出；
- Hero 在宽屏有视觉冲击但不会遮挡核心信息；
- 正文字号清晰，不能因追求编辑感而过小；
- Agent 对话内容在宽屏略向左落位，不漂浮在中央；
- 三栏宽度满足最小值和拖动边界；
- 抽屉不会遮蔽无法恢复的操作；
- 空状态与真实内容使用同一视觉系统；
- 所有页面均未出现墨绿色、青绿色或旧主题残留；
- 所有正式页面、状态和交互都能追溯到目标原型的设计语言；
- 控制台无未处理错误；
- 页面不出现假数据、私有路径或内部审核内容。

## 17. 实施阶段

设计文档审批通过后，另行编写逐步实施计划。预计分为：

1. 建立测试基线、路由骨架和设计令牌；
2. 重建全局导航与首页；
3. 重建项目目录和项目详情；
4. 建立时间线与证据页面；
5. 重建完整 Agent 工作台和本地会话；
6. 完成数据 adapter、空状态和错误降级；
7. 响应式、无障碍、动效和性能收尾；
8. 全量自动化与真实浏览器视觉验收；
9. 删除已被替换的旧页面和旧样式；
10. 同步更新 `docs/04-项目代码约束.md`，使其成为新前端架构和设计规则的长期约束。

每个阶段都应以测试先行，并保持应用可以运行。不得先删除旧前端后长期处于不可验证状态。

### 17.1 代码约束同步

全量重构不能只交付页面代码。实施完成前必须同步修订 `docs/04-项目代码约束.md`，至少补充或更新：

- 新的路由清单及页面职责；
- `pages / features / components / shared` 的依赖方向；
- 目标原型、本文档与设计令牌之间的来源优先级；
- 禁止恢复旧前端视觉、墨绿色、通用 SaaS 视觉、假数据和私有内容泄漏的规则；
- Portfolio、Project、Timeline、Evidence、Agent 的公开数据边界；
- 页面组件不得直接散布临时 API 映射的约束；
- 本地会话 7 天过期、手动清除和不上传服务端的规则；
- 响应式、键盘操作、减少动态效果和语义化要求；
- 新页面、交互和修复所需的测试与真实浏览器验证要求；
- 旧页面和旧样式删除后的禁止回引规则。

代码约束文档的修改应与最终实现一致，不能提前写入尚未落地的目录或命令。它属于本次重构的完成条件；未同步更新时，不得宣布全量重构完成。

## 18. 审批边界

用户审批本文档后，下一步只编写实施计划，不立即假设所有生产改动都已授权。实施计划应列出具体文件、迁移顺序、测试命令、浏览器验证步骤、回滚边界，以及 `docs/04-项目代码约束.md` 的同步更新节点。

在正式实施中，若真实接口能力与本文假设不一致，应优先保持事实真实性和公开边界，再通过 adapter 或诚实空状态完成页面，不得以假数据弥补后端缺口。
