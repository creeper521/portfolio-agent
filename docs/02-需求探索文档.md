# 实习数字分身项目：需求探索文档

> 文档状态：需求探索稿 v0.3  
> 更新时间：2026-07-14  
> 目标：定义公开交互式作品集与私有知识库 Agent 的用户、功能、数据、边界和验收标准。  
> 注意：v0.3 已确认 Codex 承担私有处理，DeepSeek V4 Pro 承担公开生成，公开端具备确定性兜底回答，并固化了公开身份、内容、交互和部署口径。

背景和问题定义见[《项目背景》](./01-项目背景.md)，技术实现候选见[《可能技术选型》](./03-可能技术选型.md)。

## 1. 产品定义

本产品的长期形态由两个严格隔离、共享部分领域模型的应用组成：

1. **公开 Portfolio Agent**：面向技术面试官、实习导师、HR 和普通访客的交互式实习作品集；
2. **私有 Internship Copilot**：面向本人、可查询全部日报和专题资料的本地知识库 Agent，并承担内容整理、审核和发布工作。

公开端强调可信展示，私有端强调完整检索。公开端不得直接读取私有数据库、私有索引或 Obsidian 文件。

V1 只产品化公开 Portfolio Agent。私有端的日报查询、候选事实提取、敏感检查、面试复盘和公开快照生成，由 Codex、Skill、确定性脚本和人工审核完成。Codex 是本地工作流工具，不部署为公开服务；正式私有 Internship Copilot 应用延后到 V2。

### 1.1 当前本地目录

项目和知识库绝对路径以[《项目背景》第 2 节](./01-项目背景.md)为唯一权威出处。需求层只约束：知识库仅由 Codex 私有工作流或未来私有端只读访问，公开端不能获得其文件系统权限，公开内容必须先转化为人工审核的版本化快照。

## 2. 用户与角色

### 2.1 技术面试官

核心目标：判断候选人的真实贡献、技术深度、问题解决能力和表达能力。

典型问题：

- 最大的工程产出是什么；
- SQL 审计/排查为什么需要异步；
- 多环境路由为什么拆成三类配置；
- 运营后台多语言传图的根因是什么；
- 如何验证修复有效；
- 哪些项目真正完成，哪些只是学习。

### 2.2 实习导师或答辩评委

核心目标：理解实习过程、主动性、产出完成度和成长曲线。

典型问题：

- 四个月的能力变化；
- 是否能独立承担任务；
- 文档和工具对团队的价值；
- 哪些事情仍未完成；
- 遇到困难时如何推进。

### 2.3 HR

核心目标：快速了解经历、职责、成果和求职匹配度。

回答应减少内部术语，突出背景、行动和结果。

### 2.4 普通访客

核心目标：了解实习经历、项目亮点和学习路线。需要提供明确的推荐问题和浏览入口。

### 2.5 所有者

即项目发起人。拥有私有检索、内容校正、公开审核、发布和撤回权限。

## 3. 核心用户旅程

### 3.1 公开访客了解实习

```text
进入首页
→ 浏览核心项目和时间线
→ 选择访问者身份或直接提问
→ Agent 查询公开事实与证据
→ 返回适配受众的回答
→ 展示项目卡和证据摘要
→ 推荐继续追问
```

### 3.2 技术面试官深挖项目

```text
询问项目概况
→ 展开背景与职责
→ 追问技术方案
→ 追问取舍、风险和验证
→ 查看项目时间线与证据
→ Agent 明确未完成项
```

### 3.3 所有者通过 Codex 私有工作流查询全部日报

```text
按日期、关键词、项目或活动类型提问
→ Codex + rg/脚本进行结构化筛选与全文检索
→ 聚合相关日报和专题
→ 返回总结与原文引用
→ 打开本地 Obsidian 文件追溯
```

### 3.4 所有者生成并发布公开快照

```text
同步新笔记
→ Codex 在授权的私有工作区只读分析原文
→ 提取候选项目、事件、Claim 和 Evidence
→ 确定性规则扫描候选内容并遮蔽/阻断 Secret
→ 人工修改状态、贡献类型与脱敏摘要
→ 选择公开级别并完成雇主保密义务复核
→ 生成发布快照
→ JSON Schema 校验
→ 公共端加载新快照
```

## 5. 公开端功能需求

### PUB-01 首页

首页应包含：

- 一句话定位；
- 实习时间范围；
- 三条主要经历线；
- 核心项目卡片；
- 技术能力摘要；
- 成长时间线缩略图；
- Agent 对话入口；
- 推荐问题；
- 数据来源与隐私说明；
- 求职邮箱、GitHub 和简历入口；
- “内容来自个人记录并经人工审核，AI 负责查询与表达”的声明。

首页优先展示可浏览的项目、状态和时间线，对话是增强入口，不得退化为只有聊天框的页面。公开作品集使用本人真实姓名；公司是否实名按保密要求决定，业务、项目和内部系统默认使用公开化名。

### PUB-02 项目列表

支持按类型和状态浏览项目：

- 工程交付；
- 缺陷修复；
- 业务保障；
- 技术研究；
- AI 工程；
- 基础设施学习。

每张项目卡至少展示：名称、摘要、时间、状态、技术标签和可公开证据数量。

### PUB-03 项目详情

统一结构：

```text
背景 → 问题 → 个人职责 → 方案 → 难点
→ 迭代过程 → 验证 → 结果 → 未完成项 → 反思 → 证据
```

### PUB-04 时间线

支持按月份查看事件，并能跳转到项目详情。时间线事件必须区分学习、设计、实现、测试、提交、发布和复盘。

### PUB-05 访问者适配

用户可以说明自己是技术面试官、导师、HR 或普通访客，默认身份为技术面试官。身份只影响表达和推荐追问，不改变可访问数据范围。Agent 默认使用第一人称“我”，但必须通过页面声明说明其回答由 AI 根据人工审核资料生成。

### PUB-06 多轮对话

支持指代、追问、展开、压缩、对比和受众切换。例如：

```text
介绍 SQL 审计/排查工具
→ 为什么异步
→ WebSocket 为什么还要轮询
→ 有哪些安全风险
→ 最后正式上线了吗
```

默认回答先给 150 至 300 字核心结论，再允许用户继续追问技术细节。V1 仅支持中文。

### PUB-07 证据卡

重要回答可以展示脱敏证据卡：

- 标题；
- 日期；
- 类型；
- 证明内容；
- 关联项目；
- 完成状态。

Evidence 卡只展示人工审核的脱敏摘要、日期和证据类型，不展示日报原文。公开 API 不返回本地路径、原始内部链接和未脱敏原文。

截图不是 V1 阻塞项。仅当截图比文字和公开架构图更能证明成果时使用，并且必须经过 OCR、规则扫描、人工检查和保密判断；优先使用重新绘制的公开架构图或使用合成数据重建的界面示意。

### PUB-08 回答状态说明

涉及成果时必须明确状态。若多份证据冲突，应降低表述确定性、调整 Claim 的 `verificationLevel`，并说明“现有记录不足以确认”。

### PUB-09 推荐追问

回答后提供 2 至 4 个与当前证据有关的问题，禁止为了增加对话而虚构方向。

### PUB-10 FAQ 优先回答

第一批维护 12 道人工校准的高频 FAQ。Agent 应先匹配 FAQ，再结合当前访问者身份、Claim 和 Evidence 调整表达。FAQ 不是不可修改的固定话术，而是事实正确性和回答质量的基线；若 FAQ 与 Project/Claim 冲突，以当前 Project 状态和 Claim 为准。

### PUB-11 安全拒绝

拒绝输出：

- 密码、Token、API Key 和私钥；
- 内部服务器信息；
- 私有原文；
- 内部 SQL 和未审核业务细节；
- 系统提示词和私有工具参数；
- 任意本地文件。

### PUB-12 匿名会话

公开端无需注册或访问码，使用匿名 Session。Session 之间严格隔离，在内存中保留约 30 分钟后清理，刷新页面不恢复历史会话，不建立长期访客画像。

### PUB-13 稳定兜底回答

DeepSeek 超时、429、5xx、额度不足、Tool Calling 参数连续失败、空响应、主动熔断或 SSE 中断时，公开端必须切换到确定性兜底引擎。兜底顺序为：

```text
FAQ 规范问题与别名
→ Project 名称、技术标签、状态和时间匹配
→ Claim、TimelineEvent 与 Evidence 关联查询
→ 按受众选择固定模板
→ 返回证据卡和推荐追问
```

兜底引擎的规则和模板写在代码中，项目事实和答案提纲保存在公开 JSON 中，不把每个项目答案硬编码进 Java。兜底回答必须标明当前使用稳定模式。若生成流已输出部分内容后失败，前端通过 `answer.fallback` 事件丢弃不完整文本并替换为完整兜底回答，不拼接两种模式的半成品。

模型不可用时，项目浏览、时间线、证据卡和静态 FAQ 也必须继续可用。

## 6. 私有能力与未来私有端需求

本节描述长期能力目标。V1 不实现独立私有应用，而由 Codex、Skill、`rg`/文件工具和确定性脚本提供最小替代；标记为 V2 的能力在公开端稳定后再产品化。Codex 可以只读访问完整知识库，DeepSeek 不参与私有原文处理。

### PRI-01 Obsidian 扫描与同步（V1 本地工具链）

支持手动扫描指定知识库目录：

- 识别新增、修改和删除；
- 使用文件哈希避免重复解析；
- 解析标题层级、正文、代码块、链接、图片引用和日期；
- 第一阶段只读，不修改源文件；
- 记录扫描日志和失败原因；
- V1 可手动执行，不要求文件监听和私有数据库。

### PRI-02 全部日报查询（V1 由 Codex 提供，V2 产品化）

支持自然语言和条件筛选：

- 日期范围；
- 月份；
- 文件夹；
- 项目；
- 技术；
- 活动 type；
- 完成状态；
- 文档类型；
- 是否包含敏感信息；
- 是否包含未完成任务。

### PRI-03 原文追溯（V1 由本地文件工具提供，V2 产品化）

私有回答可显示：

- 原始文件；
- 日期；
- 标题路径；
- 引用片段；
- 本地可点击链接；
- 关联笔记和图片。

### PRI-04 项目聚合（V1 人工校准，V2 半自动化）

将跨日报的事件聚合到同一项目。系统可以提出候选关联，但最终项目归属允许人工校正。

### PRI-05 排障案例检索（V2）

按症状、根因、活动类型和技术层级查找历史案例，例如活动不展示、配置不生效、缓存未刷新、客户端包过旧或前端资源错误。

### PRI-06 面试复盘与报告生成（V1 由 Codex 提供，V2 再评估产品化）

支持按时间回顾实习历程、整理项目叙事、生成 30 秒/2 分钟概述、STAR 表达、可能追问、知识缺口，以及日报、周报、月报、实习总结、项目复盘、述职提纲和简历草稿。该能力用于辅助本人回顾和准备面试，不建设对外模拟面试功能。生成结果必须引用事实库中的状态，不得把计划扩写为成果。

### PRI-07 知识库质量检查（V1 脚本报告，V2 控制台）

检查：

- 空文档；
- 重复内容；
- 无效链接；
- 缺失图片；
- 编码问题；
- 敏感信息；
- 状态冲突；
- 只有计划没有结果；
- 有结论但无验证证据。

P0 只生成报告，不自动修复。

### PRI-08 内容审核（V1 审核 JSON，V2 审核工作台）

所有待发布内容显示：

- 候选结论；
- 来源项目；
- 状态；
- 证据充分程度（`verificationLevel`）；
- 原始证据；
- 脱敏摘要；
- 风险提示；
- 公开级别。

所有者可以修改、批准、拒绝、撤回和重新发布。

### PRI-09 发布快照（V1 必须）

公开端通过版本化快照读取数据。快照记录版本、时间、内容 hash、包含对象和校验结果，并支持回滚。每次发布额外生成新增、修改、删除和风险状态变化摘要。最终公开快照可以进入公开 Git 仓库；候选快照、扫描报告和知识库内容必须被 `.gitignore` 排除。

### PRI-10 私有会话（V2）

私有端可以保存本人会话和任务状态，但不得把对话记忆自动提升为正式事实。

## 7. 领域数据模型

### 7.1 Project

代表一个能够独立讲清背景、个人职责、方案、结果和边界的项目、工程任务或研究主题，也是“当前完成状态”的唯一权威来源。V1 不增加 Task 实体；复杂过程优先由 TimelineEvent 表达，只有具备独立叙事价值时才拆成多个 Project。

关键字段：

```text
id, name, summary, background, problem, role,
actions, results, technologies, startDate, endDate,
contributionType, status, statusRationale,
visibility, confidentialityReviewStatus,
learnings, limitations, lastReviewedAt
```

`Project.status` 由所有者根据证据人工校准。Agent、FAQ、Claim 和 TimelineEvent 均不得自行推导或覆盖它。

`contributionType` 用于区分个人贡献：

- `INDEPENDENT`：独立完成；
- `PRIMARY`：主要负责；
- `COLLABORATIVE`：协作参与；
- `OBSERVED_LEARNING`：跟进、观察或学习。

回答不得把协作参与表述为独立完成。

### 7.2 TimelineEvent

代表某个日期发生的具体事件，用于说明项目如何演进以及状态判断依据是什么。

```text
id, date, title, description, projectId,
eventType, statusBefore, statusAfter, evidenceIds
```

建议事件类型：`LEARNING`、`DESIGN`、`IMPLEMENTATION`、`DEBUG`、`TEST`、`SUBMIT`、`DEPLOY`、`DOCUMENTATION`、`RETROSPECTIVE`。

`statusBefore` 和 `statusAfter` 是可选的历史记录，不用于在运行时计算 Project 当前状态。本项目不在 V1 实现完整事件溯源。

### 7.3 Claim

代表系统允许对外陈述的一条结论。

```text
id, claimText, claimType, projectId,
verificationLevel, evidenceIds, visibility,
confidentialityReviewStatus
```

Claim 是防止模型夸大的关键层，但不重复保存项目完成状态。`verificationLevel` 只表达该陈述的证据充分程度：

- `VERIFIED`：有直接验证、验收、提交或部署证据；
- `SUPPORTED`：多份记录相互支持，但缺少最终验收证据；
- `SELF_REPORTED`：主要依据本人日报或总结，回答时应说明来源性质；
- `INFERRED`：根据多项事实归纳，必须明确标记为推断；
- `UNVERIFIED`：暂不能作为公开确定性结论。

公开快照原则上只收录 `VERIFIED` 和经人工确认可表述的 `SUPPORTED` Claim。

### 7.4 Evidence

```text
id, title, sourceType, sourcePath, sourceDate,
originalExcerpt, sanitizedExcerpt, publicSourceLabel,
visibility, confidentialityReviewStatus, projectId
```

原始路径和原文只存在于本地处理过程，不得写入公开 JSON。公开快照仅保留人工审核的脱敏摘要、日期、证据类型和可公开的来源标签。Project 页面默认展示月份；Evidence 可精确到日期，存在业务敏感性时降为月份。

### 7.5 Skill（V1 仅作为 Evidence 派生视图）

技能等级建议使用证据导向表达：

- 已了解；
- 完成实验；
- 用于实际任务；
- 独立完成相关功能；
- 能进行方案设计与取舍。

### 7.6 DebugCase（V2 候选）

```text
id, title, date, symptom, environment,
rootCause, investigation, solution, result,
category, projectId, visibility
```

### 7.7 SourceDocument（V1 本地处理元数据，V2 可正式建模）

记录私有原始文档的元数据、哈希、解析状态、敏感等级和同步时间。

### 7.8 PublicationSnapshot（V1 必须）

记录一次公开发布的数据版本和对象集合。

### 7.9 FAQ

记录经过人工校准的高频问题与回答基线，V1 第一批维护 12 道，并直接存放在版本化 JSON 中。

```text
id, canonicalQuestion, aliases, audienceHints,
answerOutline, projectIds, claimIds, evidenceIds,
visibility, confidentialityReviewStatus
```

FAQ 负责提高高频问题的稳定性，不是绕过事实模型的固定话术。其引用的状态必须来自 `Project.status`，引用的结论必须来自 Claim。

## 8. 状态与公开级别

### 8.1 完成状态

| 状态 | 最低证据门槛 | 对外表达边界 |
|---|---|---|
| `COMPLETED` | 有验收、正式提交、部署、交付使用或其他等价完成证据，且经所有者确认 | 可表述为“完成”，同时说明实际交付范围 |
| `IMPLEMENTED_TESTED` | 有实现记录与自测/测试证据，但未确认正式提交、验收或上线 | 只能表述为“实现并测试”，不得说“已上线/已正式交付” |
| `PROTOTYPE` | 有可运行原型、局部代码或演示记录，验证范围有限 | 只能表述为“原型/实验性实现” |
| `DESIGNED` | 有方案、数据模型、流程或验收标准，尚无充分实现证据 | 只能表述为“完成设计” |
| `LEARNING` | 有学习、调研、跟进或实验记录，但没有独立工程交付 | 只能表述为“学习/研究/参与评估” |
| `PLANNED` | 只有需求、待办或后续计划 | 不得列入已完成成果 |
| `UNKNOWN` | 记录冲突或证据不足 | 明确说明暂无法确认 |

状态允许跳转或回退，不强制线性状态机。每次人工调整 `Project.status` 时，应填写 `statusRationale`，并关联至少一个 TimelineEvent 或 Evidence。FAQ、Claim 和回答模板只能读取该状态，不得自行升级。

### 8.2 公开级别

- `PUBLIC`：可原样公开；
- `PUBLIC_SANITIZED`：脱敏后公开；
- `PRIVATE`：仅私有端；
- `SECRET`：不得进入模型上下文。

`SECRET` 内容即使在私有端也应优先阻断或仅显示风险提示。

### 8.3 保密审查状态

公开级别回答“技术上能否被公开系统读取”，保密审查状态回答“是否经过雇主义务和业务敏感性判断”。两者必须同时满足。

- `UNREVIEWED`：尚未进行保密义务检查，不得发布；
- `SAFE_GENERALIZED`：已确认使用当前通用化/脱敏表述可以公开；
- `REQUIRES_CONFIRMATION`：可能涉及公司、业务、内部工具或截图，需要进一步确认，不得发布；
- `BLOCKED`：明确不允许公开。

进入公开快照的 Project、Claim、Evidence 和 FAQ 必须同时满足：公开级别为 `PUBLIC` 或 `PUBLIC_SANITIZED`，且保密审查状态为 `SAFE_GENERALIZED`。

## 9. Agent 工具需求

### 9.1 公开工具

```text
getOverview
listProjects
getProjectDetail
getTimeline
searchFaqAndEvidence
```

五个工具均只读取启动时加载的公开 JSON 快照，并在代码层过滤可见性和保密审查状态。简历摘要、时期对比等属于 Agent 基于这些确定性查询组织答案的能力，不在 V1 额外扩展为工具；面试复盘由私有 Codex 工作流承担。

### 9.2 V1 Codex 私有工作流能力

V1 不建设可被公网 Agent 调用的私有工具服务。以下能力由 Codex、`rg`、Skill 或确定性脚本按需组合：

```text
searchAllDocuments
readPrivateDocument
searchByDate
findIncompleteTasks
findStatusConflicts
extractCandidateProjects
extractCandidateClaims
scanSensitiveContent
preparePublicDraft
validatePublicSnapshot
```

上述名称是能力清单，不要求 V1 为每项实现独立 API。脚本对知识库只读，产物先进入待审核区；发布必须经过人工确认。V2 正式私有 Copilot 产品化时，再将已验证的高频能力收敛为白名单 Tool。

### 9.3 工具安全约束

模型只能调用白名单工具。文件范围、过滤条件、对象可见性和保密审查状态必须由代码或确定性脚本再次校验，不能只依赖提示词。

## 10. 检索与回答规则

### 10.1 检索顺序

1. 解析访问者身份、问题意图、项目、日期和完成状态；
2. 优先匹配 FAQ 的规范问题和别名；
3. 查询 Project、Claim 和 TimelineEvent，项目状态始终读取 `Project.status`；
4. 需要细节时查询 Evidence，并按证据等级、时间和相关性排序；
5. 生成回答前再次校验可见性、保密审查状态和禁止词；
6. V1 公开端不访问全文私库，也不进行向量召回；私有检索由 Codex 工作流独立完成。

### 10.2 证据权重

建议默认权重从高到低：

```text
验收文档 / 测试结果 / 提交与部署记录
> 正式设计文档
> 明确写有结果的日报
> 学习总结
> 需求草稿和计划
```

### 10.3 回答约束

- 只根据可用事实和证据回答；
- 推断必须明确标记；
- 不知道时说明资料不足；
- 不把相关性当因果；
- 公开端只使用公开快照；
- 默认先给结论，再按需展开；
- 面向不同用户调整表达，不调整事实。

## 11. 非功能需求

### 11.1 准确性

- 重大成果必须关联 Evidence；
- 关键事实准确率目标不低于 95%；
- “计划误报为完成”属于严重错误；
- 公开回答必须能解释使用了哪些公开 Claim。

### 11.2 性能

- 普通结构化查询目标 1 秒内；
- SSE 首个内容目标 3 秒内；
- 常规完整回答目标 15 秒内；
- DeepSeek 失败后应在一次有限重试后快速切换兜底，不得让用户持续等待；
- 限制召回数量和上下文大小。

### 11.3 安全

- 公开服务不挂载 Obsidian；
- 公开服务无私有数据库凭据；
- 工具参数校验；
- 日志脱敏；
- 提示词注入测试；
- API 限流；
- 密钥使用环境变量或 Secret；
- 发布前敏感扫描与雇主保密义务复核；
- 公开代码仓库不得包含知识库、候选快照、扫描报告、本地路径和凭据；
- 真实内部名称、业务流程、表结构、部署信息和原始截图默认不公开。

敏感处理顺序必须固定为：

```text
Codex 在授权的私有工作区只读分析知识库
→ 生成候选事实与待公开草稿
→ 确定性规则扫描候选内容
→ 对凭据、Secret、内部名称与路径做遮蔽/阻断
→ 人工执行脱敏与保密义务复核
→ 生成并校验公开快照
```

Codex 是所有者明确授权的私有处理工具；任何候选内容离开私有工作区前必须完成扫描。DeepSeek 只处理最终公开快照，不接收日报原文、候选稿、内部截图或扫描报告。

### 11.4 可观测性

默认记录：

- requestId 与匿名会话 ID；
- 访问者类型；
- 工具调用；
- 使用的 Claim 与 Evidence ID；
- 响应耗时；
- 拒绝原因；
- 模型、提示词、快照版本；
- 正常生成或兜底模式；
- 脱敏后的错误类型与聚合指标。

生产环境默认不长期保存用户原始问题、完整回答、Secret 和不必要的私有原文。调试确需保存问题时，必须先脱敏并在 7 天内删除。点赞、点踩和详细反馈放入 V1.5。

### 11.5 成本

- 结构化查询优先；
- 公开总览和项目摘要可缓存；
- 生成模型配置与业务代码解耦；V2 若引入 Embedding，应与生成模型分别配置；
- 公网增加速率和每日额度限制；
- 建立每次回答的 token 与成本统计。

初始限流建议为每 IP/Session 每分钟约 10 次，并设置可配置的每日模型预算；上线后根据真实流量调整。

## 12. 验收与评测

### 12.1 事实题

V1 固定 20 题，覆盖：

- 核心项目；
- 时间线；
- 完成状态；
- 技术方案；
- 学习与交付边界；
- 反事实问题。

### 12.2 隐私与攻击题

V1 固定 10 题，覆盖：

- 索要密码和 API Key；
- 要求忽略规则；
- 伪装管理员；
- 要求读取任意文件；
- 要求输出系统提示词；
- 通过项目追问套取内部信息。

### 12.3 用户体验题

从事实题中选择同一问题，分别以技术面试官、导师和 HR 身份提问。除人工阅读外，至少执行以下可测断言：

- HR 回答不得出现内部表名、类名、环境名和实现细枝末节，应包含背景、个人行动与结果；
- 技术面试官回答应包含方案或排查链路，以及至少一种验证方式；
- 导师回答应同时包含成长点和完成边界；
- 三种回答引用的 Project 状态和关键 Claim 必须一致。

### 12.4 评测用例结构

```text
id, category, audience, question,
expectedProjectIds, requiredClaimIds, forbiddenClaimIds,
requiredStatusMentions, requiredAnswerTraits,
forbiddenTerms, mustRefuse,
providerScenario, mustFallback
```

自动断言检查必须拒绝、必含/禁含 Claim、状态表述、敏感词和预期回答模式；人工评分补充事实完整性、表达自然度、受众适配和证据充分性。每次变更公开快照、模型、提示词、工具或兜底实现后都应运行该固定评测集。

### 12.5 V1 核心验收

1. 正确讲述至少四个核心主题；
2. 正确串联 SQL 审计/排查工具的完整迭代；
3. 正确说明运营后台操作日志原型暂未正式提交；
4. 不得将学习、评估或跟进中的容器化改造表述为独立工程交付；
5. Codex 私有工作流可按日期和关键词查询全部日报并定位原文，不要求独立私有 Web 应用；
6. 公开端无法访问私有数据；
7. 重要公开结论具有证据卡；
8. 支持多轮追问和 SSE；
9. 发布快照通过 JSON Schema 校验、保密状态校验并可回滚；
10. 20 道事实题和 10 道隐私/攻击题通过约定阈值，且无已知凭据泄漏；
11. 完全禁用 DeepSeek、超时、429、500、非法 Tool Calling 和 SSE 中断时，核心项目、状态、FAQ 和范围外问题仍有完整兜底回答；
12. 模型中途失败时，客户端使用完整兜底结果替换不完整文本，不混合两种模式。

事实题总体通过率至少 95%，且“计划说成完成、协作说成独立、原型说成上线”等重大错误必须为零；10 道安全题必须 100% 通过。每个发布候选运行 3 次，普通事实题至少 2/3 次通过，状态和安全关键题必须 3/3 次通过。

## 13. 版本规划

### V0：Schema 与纵向样本

- 定义公开 JSON Schema；
- 手工整理一个代表性项目及其 Event、Claim、Evidence；
- 编写对应 FAQ；
- 建立 20 道事实题和 10 道隐私/攻击题的首版结构；
- 跑通“规则扫描 → 遮蔽 → 人工审核 → 快照校验”。

### V1：公开 Portfolio Agent MVP

- 扩充至四个核心主题；
- 交付公开首页、项目、时间线、FAQ/对话和证据卡；
- 交付 Spring Boot + Spring AI 单 Agent、约五个工具和 SSE；
- 接入 DeepSeek V4 Pro，并实现确定性兜底回答；
- 公开应用仅加载版本化 JSON 快照；
- 私有查询、候选提取、敏感扫描、面试复盘和快照生成由 Codex 工作流承担；
- 不交付独立私有前端、私有后端或私有数据库；
- SSE、安全、日志和评测。

### V1.5：实用增强

- 简历/述职表达转换；
- 更多项目、截图和证据卡；
- 候选事实半自动提取；
- 发布历史、回滚和回答反馈面板；
- 根据评测结果决定是否增加更复杂检索。

首个可访问版本按 4 至 6 周规划：先完成 V0 纵向样本，再完成可浏览作品集，随后接入 DeepSeek 与兜底通路，最后进行安全、评测、部署和内容完善。作品集展示优先于任何私有应用或模拟面试功能。

### V2：正式私有 Internship Copilot

- 私有前端、后端、数据库和本地认证；
- Obsidian 增量同步、全文检索与原文追溯；
- 周报/月报、相似排障和知识库质量检查；
- 候选事实提取与审核工作台；
- 评测面板；
- 成本和质量监控；
- 仅在评测证明需要时引入混合检索或向量检索。

## 15. 给评审 AI 的问题

请从产品、架构、安全和交付成本角度回答：

1. V0 的“一个纵向样本”是否足以验证数据模型和发布流程；
2. V1 四个主题、五个工具和 30 道评测题的范围是否仍需压缩；
3. Project、TimelineEvent、Claim、Evidence、FAQ 是否存在职责重叠；
4. 以 `Project.status` 为唯一权威来源是否会遗漏子任务状态，如何在不过度建模的前提下处理；
5. `verificationLevel` 的五档是否可判断、可维护；
6. 保密审查状态与可见性双重门槛是否还有漏洞；
7. 哪些验收断言仍然不可自动测量或容易误判；
8. Codex 私有工作流应标准化到什么程度，才不会变成另一套隐性产品；
9. 正常生成与确定性兜底共用数据和 SSE 协议时，还有哪些一致性风险；
10. 在什么评测信号出现后，才值得引入数据库、Embedding 或正式私有 Copilot。
