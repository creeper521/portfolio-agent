# 实习数字分身项目：可能技术选型与架构权衡

> 文档状态：技术探索稿 v0.3  
> 更新时间：2026-07-14  
> 当前推荐：Vue 3 + Spring Boot + Spring AI + DeepSeek V4 Pro + 版本化 JSON 公开快照  
> V1 边界：只产品化公开 Portfolio Agent；私有能力由 Codex、Skill 和确定性脚本承担  
> 说明：v0.3 已锁定 Maven、Java 21、DeepSeek、单 Docker 镜像、自有服务器、Nginx/HTTPS、Codex 私有处理和确定性兜底；具体依赖小版本与发布参数在实现时填写。

背景、目录和知识资产口径以[《项目背景》](./01-项目背景.md)为准；功能、数据模型与验收标准以[《需求探索文档》](./02-需求探索文档.md)为准。本文不重复保存本地绝对路径，避免三份文档出现配置漂移。

## 1. 技术决策摘要

### 1.1 V1 推荐结论

| 领域 | V1 推荐 | 明确不做 |
|---|---|---|
| 公开前端 | Vue 3、TypeScript、Vite、Vue Router | 私有控制台、双前端工程 |
| 公开后端 | Java 21、Spring Boot 3、Spring MVC、Spring AI | 私有 Spring Boot 应用、工作流引擎、多 Agent |
| 公开数据 | 通过 JSON Schema 校验的版本化快照，启动时加载到内存 | 公网 PostgreSQL、JPA/MyBatis、pgvector |
| 检索 | FAQ、别名、结构化过滤、内存包含匹配 | P0 Embedding、中文 `tsvector`、Reranker |
| Agent | DeepSeek V4 Pro、单 Agent、约五个只读 Tool、SSE | 让公开模型直接读文件、执行 SQL 或决定公开权限 |
| 稳定兜底 | FAQ + 结构化查询 + 确定性模板，与正常通路共用公开快照 | 第二生成模型、把项目答案写死在 Java 中 |
| 私有能力 | Codex + `rg`/文件工具 + Skill/脚本 + 人工审核 | 独立私有 Web、CLI Agent、数据库、容器化服务 |
| 发布 | 私有侧生成候选 → 规则遮蔽 → 人工审核 → JSON 快照 | 公私数据库同步、自动发布原文 |
| 构建部署 | Maven；Vue 产物由 Spring Boot 托管；单 Docker 镜像；自有服务器 + Nginx + 域名 + HTTPS | 公网数据库与 Obsidian 挂载 |

### 1.2 为什么选择这条路线

V1 最有作品集价值且不能被现有本地 AI 工具替代的是：

1. 一个面试官可直接访问的交互式作品集；
2. 一套能区分“学习、设计、实现、自测、交付”的事实模型；
3. 一条经过脱敏、保密复核和人工审核的单向发布流程；
4. 可用评测证明“不会把计划说成成果、不会泄漏私有信息”的 Agent；
5. DeepSeek 不可用时，核心问题仍有完整、可追溯的稳定回答。

公开数据只有几百 KB 量级，且真正公开的部分更小。V1 使用内存快照足以支撑结构化查询，能够同时减少数据库运维、攻击面和开发工作量。Spring Boot 与 Spring AI 保留 Java 后端作品集价值；Vue 保留定制化作品集体验；确定性兜底避免公开作品集因单一模型故障失去核心问答能力。

## 2. 技术目标与约束

技术方案必须满足：

- 面向技术面试官、导师、HR 和普通访客提供公开浏览与对话；
- 公开回答只基于人工审核过的结构化事实；
- 清楚表达个人职责、完成状态、证据和未完成项；
- 支持 Tool Calling、多轮追问和 SSE 流式回答；
- DeepSeek 故障时保持完整的核心问答能力；
- 公开服务物理上无法读取 Obsidian、私有索引、原始路径和私有凭据；
- 私有原文只在本地工具链内只读处理；
- 公开内容除技术脱敏外，还要经过雇主保密义务检查；
- 能对模型、提示词、快照和工具变更做回归评测；
- 初始知识规模较小，避免用分布式系统和复杂检索掩盖数据治理问题；
- 使用真实姓名、第一人称和可验证证据建立可信表达；
- 长期可以演进为正式私有 Copilot，但 V1 不为未来假设预付全部复杂度。

## 3. 候选路线

### 3.1 路线 A：Next.js 全栈快速 MVP

可能组合：Next.js、TypeScript、静态/服务端渲染、JSON 快照、模型 SDK。

优点：

- 前后端一体，开发和部署速度快；
- 公网部署生态成熟；
- 适合快速验证作品集交互。

缺点：

- 与发起人的 Java 服务端实习方向和 Spring AI 学习积累不完全一致；
- 后端工程能力展示较弱；
- 若未来建设 Java 私有 Copilot，需要重新拆分能力。

结论：它是更快的产品验证路线，但不是当前首选。若交付时间压缩到一至两周且“先上线”优先级高于 Java 技术展示，应重新考虑。

### 3.2 路线 B：Vue + Spring Boot + Spring AI

V1 组合：Vue 3、TypeScript、Maven、Java 21、Spring Boot、Spring AI、DeepSeek V4 Pro、JSON 快照、SSE 和确定性兜底引擎。

优点：

- 与 Java 后端实习和求职方向一致；
- 能展示领域建模、Tool Calling、SSE、安全边界和评测；
- Vue 适合构建不带后台模板感的作品集界面；
- 后续可以在保留公开 API 的同时演进私有能力。

缺点：

- 比全栈 JavaScript 多一个部署单元；
- Spring AI 版本变化较快，需要锁定依赖并写适配测试；
- 如果一开始加入数据库、双应用和复杂模块，容易迅速过度设计。

结论：当前推荐。成立条件是严格遵守 V1 收缩边界：一个公开前端、一个公开后端、一个版本化数据快照。

### 3.3 路线 C：完全本地、隐私优先

可能组合：本地文件检索、本地 LLM、本地 Embedding、桌面或 localhost 界面。

优点：

- 私有知识不离开本机；
- 适合完整日报查询；
- 可以自由访问本地原文和附件。

缺点：

- 面试官无法低门槛访问；
- 不能替代公开作品集；
- 本地模型、打包与跨平台分发会引入额外工作。

结论：可作为长期私有 Copilot 的部署方向，不是公开作品集的替代方案。V1 已由 Codex 与本地文件工具覆盖主要私有价值，无需重复造应用。

### 3.4 实现起点：官方基础设施 + 自研核心 Agent

不从原始 HTTP、SSE 和 Tool Calling 协议全部手搓，也不采用 Dify、Coze、FastGPT、Flowise 或带数据库/向量库/工作流的完整 Agent 脚手架作为项目地基。

复用：

- Spring Initializr 生成标准 Maven/Spring Boot 工程；
- Spring AI 处理 ChatClient、模型流式调用和 Tool Calling 基础协议；
- Vite 生成 Vue 3 + TypeScript 工程；
- 成熟库处理 JSON Schema、测试、限流/熔断和 Markdown 安全渲染。

自行实现：

- Project、TimelineEvent、Claim、Evidence、FAQ 和状态/贡献模型；
- 五个公开 Tool、公开快照查询和引用校验；
- DeepSeek Prompt、受众适配、确定性兜底和 SSE 应用协议；
- Codex 私有工作流、敏感扫描、审核和快照发布规范；
- 20+10 评测、故障矩阵、安全边界和作品集 UI。

可以参考 Spring AI 官方示例，但只迁移已经理解的最小模式，不直接继承来源不明或包含无关基础设施的完整 Agent 模板。这样既避免重复造底层轮子，也保证作品集核心工程由本人实现并可解释。

## 4. 架构边界

### 4.1 V1 实际架构

```text
私有 Obsidian 知识库（只读）
        │
        ▼
Codex + Skill/确定性脚本
        │  私有查询、候选提取、规则扫描、Secret 遮蔽
        ▼
人工事实校准 + 雇主保密义务复核
        │
        ▼
public-snapshot.vN.json ── JSON Schema / 安全校验 / 评测
        │
        ▼
Spring Boot Portfolio App（启动时加载到内存）
        │  REST + SSE + 约五个只读 Tool
        ├──────────► DeepSeek V4 Pro 正常生成
        └──────────► FAQ + 结构化查询 + 模板稳定兜底
        ▼
Vue public-web
```

关键边界：

- 公开服务只拿到最终快照，不拿到生成快照所需的私有原文；
- 公网部署中不存在私有知识库路径、挂载、数据库账号或同步代码；
- 模型不能浏览文件系统，只能调用白名单只读 Tool；
- DeepSeek 只接收公开快照中的内容，不接收私有日报；
- Codex 是经所有者授权的私有处理工具，不属于公开网站运行时；
- 快照生成与公开运行是两个不同信任域；
- V1 不需要让私有处理工具常驻为网络服务。

### 4.2 V1 推荐目录

```text
agent/
├─ backend/
│  └─ portfolio-app/           # 单个 Spring Boot 应用，内部按包分层
├─ frontend/
│  └─ public-web/              # 单个 Vue 应用
├─ data/
│  ├─ public-snapshot.json
│  ├─ examples/
│  └─ private-workspace/        # 候选稿与扫描报告，必须 Git 忽略
├─ schema/
│  └─ public-snapshot.schema.json
├─ eval/
│  ├─ fact-cases.json
│  └─ security-cases.json
├─ scripts/                    # 本地扫描、遮蔽、校验和快照辅助脚本
├─ .gitignore                  # 排除知识库、候选快照、扫描报告与凭据
└─ docs/
```

V1 不拆 `agent-domain`、`agent-core`、`agent-infrastructure` 等独立 Maven 模块。可以在单应用内部使用 `domain`、`application`、`infrastructure`、`api` 包保持边界；只有出现第二个真实消费者后再提取共享模块。

### 4.3 长期目标架构（V2 候选，不是 V1 开发清单）

V2 若私有工具链的使用频率和痛点证明值得产品化，可增加：

```text
private-copilot-app
private-console
private-postgresql
knowledge-ingestion
publication-workbench
```

届时可以提取共享的领域枚举、快照 Schema、评测协议和 Provider 适配层。公开应用仍只读公开快照，不因私有应用出现而获得私有数据库权限。

## 5. 公开前端选型

### 5.1 推荐基础栈

- Vue 3 + Composition API；
- TypeScript；
- Vite；
- Vue Router；
- 原生 Fetch 或轻量请求封装；
- 支持 Markdown 的安全渲染；
- SSE 客户端与中断/重连处理；
- 基础单元测试与端到端测试。

V1 不使用 Pinia，组件状态与少量组合式状态足够。样式采用 CSS Variables、设计 Token 和 scoped CSS，不引入完整后台 UI 组件库；复杂无障碍交互可以按需使用少量 Headless 组件。

### 5.2 页面范围

V1 只包含：

- 首页与推荐问题；
- 项目列表/详情；
- 成长时间线；
- 对话与证据卡；
- 隐私、数据来源和状态说明；
- 求职邮箱、GitHub 和简历入口。

页面默认中文，使用真实姓名和第一人称表达，并声明内容由 AI 根据人工审核资料生成。默认访问者身份为技术面试官，支持切换导师、HR 和普通访客。完成移动端基础响应式，但主要体验面向桌面浏览和技术面试场景。

不建设私有控制台。公开数据的编辑与审核发生在 Codex、候选 JSON、辅助脚本和人工审核流程中。

### 5.3 UI 组件策略

公开作品集应优先形成个人化视觉语言，避免明显的管理后台模板感。可选择：

- 少量自建基础组件；
- Headless 组件加自定义样式；
- 只在复杂交互上引入成熟组件库。

图表库不是 P0 必需。时间线和技能证据用普通 HTML/CSS 即可；只有确有信息密度收益时再引入 ECharts。

### 5.4 SSE 客户端方案

V1 确定使用 `fetch` + `ReadableStream` 发送单个 POST 对话请求并解析 `text/event-stream`。必须处理用户主动取消、网络中断、重复事件、超时、最终完成事件和 `answer.fallback`。收到 `answer.fallback` 时，客户端清除已输出的不完整模型文本，替换为完整兜底回答。

## 6. 公开后端选型

### 6.1 Java 与 Spring

推荐：

- Maven；
- Java 21；
- Spring Boot 3；
- Spring MVC；
- Spring AI ChatClient + Tool Calling；
- Bean Validation；
- Jackson；
- 结构化日志；
- OpenAPI（可选，但有利于作品展示和联调）。

构建工具确定为 Maven。优先 Java 21，因为它是长期支持版本并可展示现代 Java 能力。Spring Boot 与 Spring AI 的具体小版本在开始实现时选择相互兼容的稳定组合并锁定；只有部署环境或兼容性明确要求时才降至 Java 17。

V1 没有业务数据库，因此不需要 Spring Data JPA、MyBatis、Flyway 和数据库连接池。V2 若引入 PostgreSQL，可优先 JPA 处理普通实体，复杂检索使用原生 SQL，避免同时维护多套 ORM。

### 6.2 应用内分层

建议在单个 Spring Boot 应用内部保持四层：

- `domain`：Project、TimelineEvent、Claim、Evidence、FAQ、状态枚举；
- `application`：查询服务、回答编排、快照加载和校验；
- `infrastructure`：模型 Provider、JSON 读取、日志与限流适配；
- `api`：浏览 API、对话 API、SSE DTO 和错误处理。

这是代码组织，不是四个独立构建模块。

### 6.3 单 Agent 职责

Agent 负责：

- 识别访问者类型与问题意图；
- 选择只读工具；
- 根据查询结果组织答案；
- 调整表达深度而不改变事实；
- 明确不确定性、完成边界和拒绝原因；
- 推荐与当前证据相关的追问。

Agent 不负责：

- 决定 Project 当前状态；
- 修改 Claim 或 Evidence；
- 决定内容是否允许公开；
- 读取任意文件、执行 SQL 或调用网络；
- 把会话内容写回事实库。

公开 Agent 使用 DeepSeek V4 Pro。默认 temperature 为 0 至 0.3，以稳定事实表达。V1 不做自动双 Provider 切换，但 Provider 配置和业务代码保持解耦。

### 6.4 V1 Tool 集

建议收敛为五个工具：

```text
getOverview
listProjects
getProjectDetail
getTimeline
searchFaqAndEvidence
```

工具返回稳定、可验证的 DTO，并在代码中只选择满足以下条件的对象：

```text
visibility in [PUBLIC, PUBLIC_SANITIZED]
AND confidentialityReviewStatus == SAFE_GENERALIZED
```

不要把“生成简历”“比较时期”等表达方式各自做成 Tool；面试复盘由私有 Codex 工作流承担。工具数量越少，越容易测试参数、权限和跨模型兼容性。

### 6.5 SSE 事件协议

V1 可使用最小事件集：

```text
session.created
answer.delta
evidence.added
suggestion.added
answer.completed
answer.fallback
error
```

工具名、内部参数、原始模型思维过程和私有检索过程不应返回给公开客户端。可将 `projectIds`、`claimIds`、`evidenceIds` 和 `answerMode` 作为完成事件中的可审计元数据，但只允许引用公开快照对象。

### 6.6 确定性兜底引擎

兜底引擎与 DeepSeek 共用同一公开快照和查询服务，不引入第二个生成模型。触发条件包括：连接超时、429、5xx、额度不足、Tool Calling 参数连续校验失败、空响应、熔断器开启、主动禁用 Provider 和 SSE 中途异常。

执行顺序：

```text
FAQ 规范问题/别名匹配
→ Project 名称、标签、状态和时间匹配
→ Claim、TimelineEvent、Evidence 关联查询
→ 按技术面试官/导师/HR/普通访客选择固定模板
→ 返回 Evidence 卡与 2～4 个推荐追问
```

Java 中只保存规范化、匹配、受众选择和模板组装规则；事实、FAQ 答案提纲、项目状态和证据仍保存在 JSON 快照。普通 Provider 错误最多有限重试一次，随后快速兜底。模型在首段内容前失败时直接返回兜底；已经流式输出后失败时发送 `answer.fallback`，由前端整体替换未完成回答。

可使用 Resilience4j 或等价的轻量机制实现超时、有限重试和熔断，但不得为了该能力引入完整工作流平台。

## 7. 公开数据与快照

### 7.1 为什么 V1 不使用 PostgreSQL

- 公开数据量小，内存读取足够；
- 数据更新频率低，适合审核后整体发布；
- 不存在在线编辑和复杂事务；
- JSON 文件天然适合版本审查、Schema 校验和回滚；
- 删除公网数据库可减少成本、凭据、端口和注入攻击面；
- 公开端只接触脱敏快照，隔离语义更清晰。

因此，V1 不使用 PostgreSQL、JPA、MyBatis、Redis 或 pgvector。

### 7.2 快照内容

快照至少包含：

```text
snapshotVersion
schemaVersion
generatedAt
contentHash
changeSummary
projects[]
timelineEvents[]
claims[]
evidence[]
faqs[]
```

不得包含：

- 本地绝对路径；
- 原始文件名或可推断内部目录的相对路径；
- 未脱敏原文；
- 内部 URL、IP、端口、账号和环境名；
- 任何 Secret；
- `PRIVATE`、`SECRET`、`UNREVIEWED`、`REQUIRES_CONFIRMATION` 或 `BLOCKED` 对象。

### 7.3 加载与回滚

应用启动时执行：

1. JSON 语法与 Schema 校验；
2. 引用完整性校验；
3. 可见性与保密审查状态校验；
4. ID 唯一性和状态枚举校验；
5. 计算并记录快照 hash；
6. 构建只读 Map/List 索引。

任一强校验失败时拒绝启动，避免带病发布。V1 回滚方式是恢复上一个已通过评测的快照并重新部署/重启。每次发布生成新增、修改、删除和风险状态变化摘要。最终公开快照跟随代码和部署制品进入公开 Git 仓库；候选文件、扫描报告、知识库、路径和凭据必须被 `.gitignore` 排除。

### 7.4 是否需要热更新

P0 不需要。作品集数据更新频率低，重新部署更容易审计。如果后续发布频率明显升高，可增加管理端点触发原子重载，但仍应只接受预先校验过的本地部署制品，不接受公网任意上传。

## 8. 检索方案

### 8.1 V1 公开检索

公开端优先级如下：

1. FAQ 规范问题、别名和关键词匹配；
2. Project ID、名称别名、标签和状态过滤；
3. Claim 与 Evidence 的结构化关联；
4. 对已公开的短文本做规范化后包含匹配；
5. Agent 根据返回结果组织答案。

中文规范化可包含：大小写统一、全半角统一、空白与标点处理、常用项目别名映射。V1 数据集很小，不需要通用搜索引擎。

### 8.2 PostgreSQL 原生全文检索的中文限制

PostgreSQL 原生 `tsvector`/默认 parser 不提供适用的中文分词能力，不能把它直接视为可靠的中文全文检索方案。若 V2 引入 PostgreSQL，应通过真实评测选择：

- `ILIKE` + 结构化过滤：数据小、实现最简单；
- `pg_trgm`：适合模糊匹配，部署相对简单；
- `pg_bigm`：二元组对中日韩文本更友好，但需要额外扩展；
- `zhparser`、`pg_jieba` 或应用层分词：中文召回更强，但镜像、词典和运维成本更高；
- 独立搜索引擎：只有规模、并发和搜索需求明确增长后再评估。

不能仅因为 PostgreSQL “内置全文检索”就在 P0 使用 `tsvector`。

### 8.3 Embedding 与 pgvector

P0 明确不使用 Embedding。引入向量检索必须先出现可量化信号，例如：

- 20 道事实题中，多个问题因同义表达稳定漏召回；
- FAQ 和别名维护成本开始明显上升；
- 相似排障案例成为高频需求；
- 私有文档规模显著增加，文件/关键词检索无法满足；
- 离线评测证明混合检索相对基线有明确提升。

即使引入 pgvector，也应保留 Project/Claim/Evidence 的结构化过滤和状态权威来源。向量相似度不能决定“是否完成”和“是否可以公开”。

### 8.4 Rerank

V1 使用确定性排序：FAQ 精确度、项目关联、证据等级、日期和人工优先级。只有候选数量和评测结果证明需要时才引入模型 Reranker。

## 9. V1 私有工具链

### 9.1 定位

私有工具链是公开快照的本地生产线，也是正式私有 Copilot 的需求验证手段；它不是另一个需要登录、部署和长期运行的产品。Codex 是指定的私有处理工具。

V1 可组合：

- Codex：读取完整本地知识库，执行自然语言查询、跨文档总结、候选提取和面试复盘；
- `rg`/文件读取：日期、关键词、项目和原文定位；
- 确定性脚本：统计、链接检查、Secret 扫描、JSON 生成与 Schema 校验；
- Skill：固化“如何扫描、如何归纳、如何审核”的重复工作流；
- 人工审核：状态、个人贡献、脱敏表述和保密义务最终裁决。

### 9.2 路径配置

知识库绝对路径以背景文档为准，不写入业务代码和公开仓库。若脚本需要配置，使用仅存在于本机的环境变量，例如：

```text
PRIVATE_KNOWLEDGE_BASE_PATH=<local-obsidian-path>
```

脚本应校验解析后的根路径，只允许在该根目录内只读访问，不能接受模型生成的任意路径。V1 不需要把知识库挂载进 Docker。

### 9.3 V1 能力边界

应支持：

- 按日期和关键词查询全部日报；
- 定位原始文件与引用片段；
- 发现计划/完成状态冲突；
- 生成 Project、Event、Claim、Evidence、FAQ 候选；
- 生成敏感扫描和保密待确认清单；
- 生成并验证公开 JSON 草稿。

面试复盘由 Codex 私有工作流完成，包括按时间回顾、项目叙事整理、30 秒/2 分钟概述、STAR 表达、可能追问和知识缺口，不在公开网站建设模拟面试功能。

不要求：

- 私有多轮会话持久化；
- 私有数据库；
- 文件监听与实时增量索引；
- Web 审核工作台；
- 自动写回 Obsidian；
- 私有服务容器化。

## 10. 敏感信息与保密审查

### 10.1 固定处理顺序

```text
Codex 在授权的本地工作区只读分析私有知识库
→ 生成候选事实与待公开草稿
→ 确定性规则扫描候选内容
→ 阻断或遮蔽 Secret、内部名称与路径
→ 人工脱敏与雇主保密义务复核
→ 生成公开候选
→ Schema、安全规则与评测
→ 发布
```

该顺序是硬约束，不是建议。Codex 是所有者明确授权的私有处理工具，可以读取本地原文；任何候选内容离开私有工作区或进入公开快照前，必须经过确定性扫描和人工复核。DeepSeek 不参与私有知识处理，也不能读取未审核草稿。

### 10.2 确定性扫描范围

- API Key、Token、私钥和密码模式；
- IP、端口、内部域名和 URL；
- 数据库、SSH、云平台和 CI/CD 配置；
- 账号、玩家/测试标识和邮箱；
- Windows/Linux 内部路径；
- 自定义项目名、环境名、业务名和公司黑名单；
- 截图文件名及 OCR 后文本（若 V1 发布截图）。

规则扫描会有误报和漏报，因此不能替代人工审查。

### 10.3 保密状态

Project、Claim、Evidence 和 FAQ 都使用：

```text
UNREVIEWED
SAFE_GENERALIZED
REQUIRES_CONFIRMATION
BLOCKED
```

只有 `SAFE_GENERALIZED` 可以进入公开快照。这里审查的不只是凭据，还包括内部系统命名、业务流程、工具能力、表结构、部署方式、截图和可能受保密协议约束的架构细节。必要时应征求导师或公司确认；无法确认时宁可通用化或不发布。

### 10.4 模型数据边界

- Codex 是私有处理工具，可在授权工作区只读访问完整知识库；
- DeepSeek V4 Pro 只接收人工审核后的公开快照数据和公开用户问题；
- 不通过 DeepSeek 处理日报原文、候选快照、内部截图或扫描报告；
- DeepSeek 输出仍需做敏感词、公开对象引用 ID 和状态一致性校验；
- Codex、知识库路径和私有扫描配置不得进入公开服务器运行时。

## 11. 模型与 Provider

### 11.1 Provider 抽象

配置外置：

```text
baseUrl
modelName
apiKeyEnvironmentName
temperature
maxTokens
timeout
```

公开生成模型确定为用户提供的 DeepSeek V4 Pro API。实际配置使用环境变量，例如：

```text
DEEPSEEK_BASE_URL
DEEPSEEK_MODEL
DEEPSEEK_API_KEY
```

业务代码依赖 Spring AI 抽象和项目内部适配层，不依赖 DeepSeek 专有响应对象。若接口兼容 OpenAI 协议，优先使用相应的 Spring AI 兼容配置。依赖版本应锁定，并用 Tool Calling 与 SSE 契约测试防止升级破坏。API Key 不在聊天、Git、镜像或前端中出现。

### 11.2 DeepSeek 接入验收

- 中文理解与自然表达；
- Tool Calling 成功率和参数 Schema 遵循；
- 对“证据不足”和安全拒绝的指令遵循；
- 流式输出稳定性；
- 延迟、成本和限额；
- Spring AI 兼容性；
- 数据使用、保留和合规条款。

DeepSeek 已确定为 V1 Provider，但只有普通中文问答、流式输出、Tool Calling、流式 Tool 参数、多轮消息、超时和限额测试通过后才能上线。固定的 20 道事实题、10 道攻击题和兜底故障测试用于决定是否达到发布标准，不再用于 V1 重新选择供应商。

### 11.3 Prompt 策略

系统提示词应简洁地规定：事实来源、状态权威来源、受众适配、必须拒绝事项和回答结构。不要把完整知识库、所有安全逻辑或大量固定答案堆进 Prompt；FAQ 和事实通过 Tool 返回，权限由代码执行。

## 12. 安全设计

### 12.1 公开运行时

- 无访问码的匿名只读访问；
- 请求体、字段长度、会话轮数和 Tool 参数限制；
- IP/会话级限流与全局每日预算；
- TLS 与反向代理安全头；
- CORS 仅允许公开前端域名；
- 公网容器/进程无私有文件挂载和私有环境变量；
- 日志不记录 Secret、完整 Prompt 或不必要的用户原文；
- DeepSeek 超时、取消、熔断和确定性兜底；
- 依赖与镜像漏洞扫描。

### 12.2 Prompt Injection

安全不能依赖“请不要泄漏”的提示词。必须通过：

- 五个只读白名单 Tool；
- Tool 参数 JSON Schema 与服务端二次校验；
- 不提供文件、Shell、数据库和任意 URL 工具；
- 工具返回前过滤可见性与保密状态；
- 输出禁止词与引用 ID 校验；
- 固定攻击评测集；
- 限制会话上下文，避免把用户输入当作系统事实。

由于公开快照不包含私有数据，即使模型被诱导忽略规则，也没有读取私库的能力。这是最重要的安全保证。

### 12.3 会话与数据保留

V1 使用内存中的短期匿名 Session，只服务于多轮指代，约 30 分钟后过期，刷新页面不恢复历史会话，不建立长期访客画像。生产环境默认不长期保存用户原始问题和完整回答，只记录公开对象 ID、耗时、模式和聚合指标；调试确需保存问题时先脱敏，并在 7 天内删除。

## 13. 可观测性与成本

### 13.1 建议记录

- requestId、匿名 sessionId；
- audience、意图分类；
- Tool 名称与成功/失败，不记录敏感参数；
- 返回的公开 Project/Claim/Evidence ID；
- 模型、提示词版本和快照版本；
- 首段延迟、总时长、token 与估算成本；
- 拒绝原因和错误类型；
- `answerMode`（DeepSeek 正常生成或确定性兜底）；
- 熔断、重试、限流和兜底触发原因。

### 13.2 V1 指标

- Tool 调用成功率；
- 无答案率；
- 证据引用覆盖率；
- 事实题通过率；
- 攻击题拒绝/脱敏通过率；
- SSE 首段延迟与完整响应时间；
- 单次回答成本；
- 限流和预算触发次数。

V1 使用结构化日志和离线评测报告即可，不需要立即接入完整监控平台。总览和固定 FAQ 可缓存，但缓存键必须包含快照、模型和提示词版本。初始建议每 IP/Session 每分钟约 10 次，并设置每日模型预算；参数上线后按真实流量调整。

## 14. 测试与评测

### 14.1 确定性测试

- JSON Schema、枚举和引用完整性；
- 快照中不得出现私有级别或未通过保密审查的对象；
- SnapshotRepository 内存查询；
- 五个 Tool 的参数边界与结果过滤；
- 状态始终来自 Project；
- REST/SSE 协议、中断和异常；
- FAQ、项目、状态、受众模板和范围外问题的确定性兜底；
- DeepSeek 禁用、超时、429、500、空响应、非法 Tool Calling 和流式中断；
- `answer.fallback` 能整体替换不完整回答；
- 前端项目浏览、对话和证据卡；
- Secret 扫描器的正反样本；
- 发布旧快照的回滚测试。

### 14.2 Agent 评测

V1 使用 20 道事实题和 10 道隐私/攻击题。用例结构与受众断言以需求文档第 12 节为准。执行时同时保存：

- 快照版本；
- 模型与参数；
- 提示词版本；
- Tool 调用轨迹；
- 引用的公开对象 ID；
- 自动断言结果；
- 必要的人工评分。

每个发布候选运行 3 次。普通事实题至少 2/3 次通过，状态和安全关键题必须 3/3 次通过。不要用单次自然语言完全匹配判断好坏，优先断言 Claim、状态、拒绝、禁止词、必要答案特征和预期回答模式。

### 14.3 发布门禁

建议至少满足：

1. 所有 Schema、安全和引用完整性测试通过；
2. 事实题不得出现“计划/学习误报为完成”的严重错误；
3. 攻击题不得泄漏禁止内容；
4. 四个核心主题均有可解释的 Claim 与 Evidence；
5. 人工抽查 HR、技术面试官和导师三类回答；
6. 上一个稳定快照可立即恢复；
7. 事实题总体通过率至少 95%，且计划说成完成、协作说成独立、原型说成上线等重大错误为零；
8. 10 道攻击题 100% 通过；
9. DeepSeek 故障矩阵下核心问题仍返回完整兜底答案。

发布报告同时记录正常生成与兜底模式结果，避免只验证模型正常时的理想路径。

## 15. 部署方案

### 15.1 公网组件

```text
Nginx / 域名 / HTTPS
        │
        ▼
单 Docker 镜像
└─ Spring Boot portfolio-app
   ├─ 托管 Vue public-web 构建产物
   └─ 内置 public-snapshot.json
```

部署到用户已有服务器，域名由用户购买。V1 不拆分前后端部署，不包含 PostgreSQL、pgvector、Redis、私有应用或 Obsidian 挂载。Docker 镜像只包含公开应用、前端静态资源和公开快照。

### 15.2 Secret 管理

模型 API Key 由服务器环境变量或受控 Secret 文件注入，不写入 Git、Docker 镜像、前端构建产物或快照。日志不得打印请求头和完整 Provider 配置。

### 15.3 私有侧

V1 直接在本机运行 Codex、Skill 和脚本，不建设私有 CLI Agent、容器或服务。Codex 可以只读处理完整知识库；候选数据和扫描报告保存在被 Git 忽略的非公开目录；只有最终通过规则扫描、人工审核和评测的快照进入公开部署输入。

## 16. 技术演进顺序

### V0：不接 LLM 的纵向数据样本

- 定义 JSON Schema；
- 手工录入一个代表性项目；
- 建立对应 TimelineEvent、Claim、Evidence 和 FAQ；
- 实现快照校验与内存查询；
- 验证它能回答典型面试问题；

- Spring Boot 浏览 API；
- 扩充到四个核心主题；
- 完成移动端基础响应式与可访问性。

### V1-B：单 Agent 对话

- Spring AI 与 DeepSeek V4 Pro 适配；
- 五个只读 Tool；
- SSE、多轮 Session 与受众适配；
- FAQ + 结构化查询 + 确定性模板兜底；
- 20+10 评测、限流、日志和预算；
- 公网部署与回滚演练。

### V1.5：基于反馈增强

- 更多项目、FAQ、截图和证据卡；
- 简历/述职表达与作品集展示增强；
- 半自动候选提取；
- 发布历史和回答反馈；
- 只有评测证明需要时才升级检索。

首个可访问版本按 4 至 6 周规划：第 1 周完成 V0 纵向样本；第 2 周完成主要浏览页面和四个主题数据；第 3 至 4 周完成 DeepSeek、Tool Calling、SSE 与兜底通路；第 5 至 6 周用于安全、评测、部署、内容审核和视觉完善。实际以可用时间和数据审核进度调整，不为赶进度降低保密门槛。

### V2：私有 Copilot 产品化门槛

只有出现以下信号才启动：

- 本地工具链被高频重复使用且操作成本明显；
- 多次需要增量同步、长期会话或审核工作台；
- 文件检索无法稳定支撑全部日报查询；
- 正式私有应用能形成新的作品集或长期生产力价值。

届时再评估 PostgreSQL、JPA、中文搜索扩展、Embedding、私有前端和本地认证。

## 17. 主要技术风险与应对

| 风险 | 影响 | 当前应对 |
|---|---|---|
| V1 再次膨胀成双应用 | 延期、无法形成可展示闭环 | 以本表和需求 P0 为门禁，私有应用固定延后 V2 |
| 事实模型维护成本被低估 | 状态漂移、回答失真 | Project.status 唯一权威；V1 手工校准；评测回归 |
| DeepSeek Tool Calling 不稳定 | 参数错误、回答失败 | 五个小 Tool、严格 Schema、契约测试和确定性兜底 |
| DeepSeek 超时、限流或不可用 | 公开问答中断 | 有限重试、熔断、FAQ + 结构化查询 + 确定性模板兜底 |
| Spring AI 快速变化 | 升级破坏接口 | 锁版本、封装适配层、避免追最新特性 |
| 自动脱敏漏检 | 泄密与合规风险 | 候选内容确定性扫描、人工保密复核、公开端无私库 |
| 公开项目本身涉及保密义务 | 即使无凭据仍不可发布 | 通用化命名、四态审查、必要时征求确认 |
| JSON 快照人工编辑出错 | 引用断裂、状态冲突 | Schema、引用校验、评测、版本回滚 |
| 中文问法漏召回 | 回答不完整 | FAQ、别名、结构化匹配；用评测决定是否升级 |
| 公网模型成本或滥用 | 超预算、服务不可用 | 限流、每日预算、缓存、超时与降级 |
| 只有聊天、作品集价值不明显 | 面试官无法快速判断产出 | 浏览页面优先，Agent 是增强而不是唯一入口 |
